# client — Engineering Insights

Append-only log of non-obvious, hard-won lessons for this module (`@devdigest/web`).
Managed by the `engineering-insights` skill. Add each entry under one section; keep
it actionable cold; never edit or delete existing entries.

## What Works

- This module had no precedent for a **pure hook-level unit test** (all existing mutation tests mock the whole `@/lib/hooks/*` module and assert on a component, e.g. `SkillsTab.test.tsx`). To unit-test a hook itself (e.g. `useSetAgentDocuments` in `lib/hooks/documents.test.tsx`), mock `../api` (not the hooks module) with `vi.mock("../api", () => ({ api: { get: ..., post: ... } }))`, then drive it with `@testing-library/react`'s `renderHook(() => useX(id), { wrapper })` where `wrapper` mounts a fresh `new QueryClient()` inside `<QueryClientProvider>` — every `useQuery`/`useMutation` throws "No QueryClient set" without one, even for a query that's `enabled:false`. Assert cache writes via `qc.getQueryData([...key])` after `await waitFor(() => expect(result.current.isSuccess).toBe(true))`. Two gotchas: (a) this file needs a `.tsx` extension, not `.ts` — the wrapper JSX (`<QueryClientProvider>...</QueryClientProvider>`) fails to parse under esbuild's `.ts` loader (no JSX in `.ts`), even though vitest's `include` glob (`src/**/*.test.{ts,tsx}`) matches both; (b) give each test its own `QueryClient` instance (don't share one across `it()` blocks) or a cache write from one test leaks into the next's assertions.

## What Doesn't Work

- The BlastRadius panel (`pulls/[number]/_components/BlastRadius/BlastRadius.tsx`) originally computed `isEmpty = !data || totals.symbols === 0 || !hasCallers`, where `hasCallers = data.symbols.some(g => g.callers.length > 0)`. The `|| !hasCallers` term COLLAPSES THE WHOLE PANEL to the one-line `noDownstream` message whenever NO changed symbol has a resolved downstream caller — so a PR with 53 changed symbols but 0 callers renders as just text, with every symbol (and the Tree/Graph tree) hidden. This came straight from the plan (`docs/plans/blast-radius.md:218` "when `totals.symbols === 0` or no callers → explicit empty state"), which contradicts the plan's OWN acceptance criterion ("empty state when `totals.symbols === 0`", lines 260/296). "0 callers" is common and expected: `getResolvedCallers` (`server/.../repo-intel/repository.ts`) is precision-over-recall — it only counts `references` rows whose resolved `declFile` ∈ changed files (NULL `declFile` dropped), so re-exported/barrel symbols or unresolved refs yield 0 callers even on a full index (no degraded badge shown). Fix: gate empty ONLY on symbols (`isEmpty = !data || totals.symbols === 0`); when `totals.symbols > 0 && totals.callers === 0`, still render the symbol tree and show the `noDownstream` line as a NOTE above it (`s.noCallersNote`), and add a distinct `blast.empty` ("No impacted symbols found for this PR.") for the true-zero case so the count-0 `noDownstream` string isn't shown. `SymbolRow` already renders fine with `callers: []` (the caller `<ul>` is conditional), and `BlastGraph` already guards `hasContent` → its own empty message. Regression test: a fixture with symbols but `callers: []` must assert the symbol NAMES render (not the empty state).
- The Skill ConfigTab (`skills/[id]/_components/SkillEditor/_components/ConfigTab/ConfigTab.tsx`) must PATCH only the fields that actually changed — sending the whole form on every save DEFEATS the server's body-only versioning. The server versions a skill by its BODY only (`server/src/modules/skills/{helpers.ts:isBodyChange, repository.ts:update}`; `skill_versions` stores body snapshots): a DRAFT (v0) commits to v1 on the first save that *includes a body*, and a committed skill bumps + snapshots ONLY on a real body change — name/description/type/enabled edits never version. The original ConfigTab always sent `{name, description, type, body, enabled}`, so a metadata-only save on a draft resent the unchanged scaffold body and prematurely committed v1 (burning a version on the scaffold the server explicitly tries to avoid). This is also why "edit something, save → always v1" looks like a bug: metadata edits legitimately don't version, and the always-full-patch hid that. Fix: build `patch` from per-field diffs vs the `skill` prop (`if (name !== skill.name) patch.name = name; …`) and send only changed keys; gate the Save button with an `isDirty` flag (`disabled={update.isPending || !isDirty || !name || !body}`) so it's off until something changes. Test with `fireEvent.change` (no user-event in this package) then assert `mutate` got `{ patch: { <onlyChangedField> } }`; a no-change render must leave Save `toBeDisabled()`.
- Linking a finding's `file:line` to GitHub via `githubBlobUrl` (`/{repo}/blob/{sha}/{file}#L{line}`) 404s in two real cases: (a) FORK PRs — `pr.head.sha` lives on the contributor's fork, not the base repo, so `base-repo/blob/{forkSha}` doesn't exist; (b) the demo SEED — `seed.ts` uses a fabricated repo (`acme/payments-api`) + fake `head_sha` (`a1b2c3d4e5f6`), so EVERY github.com link 404s regardless of format (a user reported "always 404" here; the format was fine, the data isn't real). Fix: link into the PR's Files-changed diff with `githubPrFileUrl` (`/{repo}/pull/{number}/files#diff-<sha256(path)>R{line}` in `lib/github-urls.ts`) — the Files view is owned by the BASE repo, so it resolves for fork PRs. The `#diff-` hash is the SHA-256 hex of the repo-relative path, computed async via Web Crypto in the `usePathShas` hook (`lib/hooks/use-path-shas.ts`); render the bare `/pull/{n}/files` URL immediately and upgrade once the hash resolves (and when `crypto.subtle` is missing — jsdom/insecure context — keep the bare URL). `R{line}` targets the added/right side; it only anchors precisely if the line is in the diff, but the file still loads otherwise. To verify a real (non-404) link you MUST import a real public repo — the seed can't resolve.
- The vendored `Checkbox` (`src/vendor/ui/kit/Checkbox.tsx`, a `<button>` inside a `<label>`) fires `onChange` **TWICE** per physical click (the label re-dispatches activation to its labeled control). Beyond the server-side concurrency this causes (`server/INSIGHTS.md` advisory-lock note), it also produces a LOGICAL toggle add-back in `AgentEditor/_components/SkillsTab/SkillsTab.tsx`: its `toggle(id, on)` trusts the `on` (`!checked`) the checkbox passes and closes over `attached`; when React flushes a re-render BETWEEN the two synthetic clicks, the second click reads the now-flipped `checked` and computes the OPPOSITE intent → it re-adds the skill you just removed, so disabling a skill silently leaves it enabled. It's a RACE: small lists (1–2 linked skills) render fast enough to flush between the two clicks → reproduces ~1-in-5; larger lists almost never do → "works 80% of the time". The two `POST /agents/:id/skills` carry DIFFERENT `skill_ids` (`[…without id]` then `[…with id]`), so the server advisory lock can't fix it — last writer wins. FIX (vendor is "do not touch", so client-side): a per-skill-id in-flight guard `const toggling = React.useRef<Set<string>>(new Set())` — `toggle` returns early if `toggling.current.has(id)` (drops the spurious synchronous second fire), else adds the id and clears it via the mutation's `onSettled` (`setSkills.mutate(ids, { onSettled })`). The second fire is dispatched synchronously within the same click gesture, always inside the network round-trip window, so it's reliably ignored; `onDrop`/reorder is keyed differently and passes no `onSettled` so it's unaffected. Regression test (`SkillsTab.test.tsx`): `fireEvent.click` the same checkbox TWICE back-to-back, assert `setMutate` called ONCE with `[]` (verified it fails — 2 calls — without the guard). A single `await`-each / single-click test proves nothing.
- A `position: fixed` portal popover positioned from a ONE-TIME `getBoundingClientRect()` (e.g. `FindingsHoverCard`) DETACHES on scroll: `fixed` pins it to the viewport, so when the page/timeline scrolls the trigger moves but the card stays floating over unrelated content. Must dismiss (or reposition) on `scroll`+`resize` while open. Two subtleties: (a) register the `scroll` listener in the CAPTURE phase (`window.addEventListener('scroll', fn, true)`) — `scroll` does NOT bubble, and the content may live in a NESTED `overflow:auto` scroller a non-capture window listener would miss; (b) IGNORE scrolls originating inside the popover's own `overflow:auto` list via `popoverRef.current?.contains(e.target)`, or scrolling a long findings list self-dismisses the card. Regression test: open via `mouseEnter`, `fireEvent.scroll(window)`, assert it's gone (`RunHistory.test.tsx`).
- A master/detail page whose LIST is its own `overflow:auto` scroll container (e.g. the Skills sidebar in `skills/_components/SkillsWorkspace/SkillsWorkspace.tsx`) JUMPS that list back to the top whenever you select a row, because Next.js App Router's `router.push`/`router.replace` default to `scroll: true`, which scrolls the nearest scroll container — NOT just `window` — to the top on every navigation. Symptom a user will report: "scroll down, click the last skill, the list snaps back to the top." Fix: pass `{ scroll: false }` to EVERY intra master/detail navigation — the row-select push (`router.push(`/skills/${id}?tab=${tab}`, { scroll: false })`) AND the tab-change `router.replace`s in BOTH `skills/page.tsx` and `skills/[id]/page.tsx` (a tab switch shouldn't move the list either). There's no visible window scrollbar to hint at this — the layout is `height: calc(100vh - 52px)` with inner scrollers, so the reset only ever hits the inner container.
- `BlastGraph` (`BlastRadius/BlastGraph.tsx`) early-returns its "No downstream callers to graph." message when `callers.length === 0 && rightNodes.length === 0`, so a PR with N changed symbols but 0 resolved callers/endpoints renders a BLANK graph and the Tree→Graph toggle LOOKS broken (the toggle itself works — it's the empty gate). 0 callers is common/expected for route-handler PRs (see the `getResolvedCallers` precision-over-recall note above). Fix: draw the left changed-symbol column whenever `symbols.length > 0`, and only show the empty copy when there are literally no symbols.
- A PR-score gauge already exists — do NOT build a new one when asked to add a "score badge" to the Overview/Intent area. `_components/VerdictBanner/VerdictBanner.tsx` renders the verdict pill + summary + `CircularScore` "PR SCORE" and self-guards `score != null`; `CircularScore` is a `vendor/ui` primitive. It's currently mounted ONLY inside `ReviewRunAccordion` (Findings tab), NOT on the Overview — but the design places it atop the PR Brief. To wire it: pass the latest review (`reviews[0]` from `usePrReviews`, already fetched in `pulls/[number]/page.tsx`; `verdict`/`summary`/`score` per `vendor/shared/contracts/review-api.ts`) into `OverviewTab` and render `<VerdictBanner>` above `IntentCard`. No new endpoint/contract needed.
- Blast-radius caller links go silently NON-CLICKABLE in BOTH the tree (`BlastRadius.tsx` `CallerItem`) and graph (`BlastGraph.tsx`) views when the repo-intel index has no recorded SHA. Both build the href from `blastCallerUrl(repoFullName, indexedSha, file, line)` (`lib/github-urls.ts`), which originally returned `null` unless BOTH inputs were present — and `indexedSha` (= `data.index.lastIndexedSha`) is `null` at runtime for an un-indexed repo / older index row / empty-string sha, EVEN WHEN symbols and callers resolve fine. Result: the panel renders plain `<span>`s and clicks do nothing. `repoFullName` is rarely the culprit (it also powers the PR's GitHub link and the page title, so you'd notice it missing); the silent null is almost always `indexedSha`. Fix: `blastCallerUrl` now returns `null` ONLY when `repoFullName` is missing and falls back to `HEAD` for the blob ref when the sha is absent (`githubBlobUrl(repoFullName, indexedSha || "HEAD", file, line)`), so the caller stays clickable. TRADEOFF: a HEAD-pinned line number can drift if the default branch moved past the indexed commit; when `lastIndexedSha` IS present the link stays sha-pinned and accurate. (Distinct from the finding-link 404 entry above: that's a wrong-URL for findings; this is a no-URL for blast callers. Both views share the one helper, so fix it there, not per-view.)
- The PR-Brief two-column grid (`OverviewTab/styles.ts` `briefGrid` — Intent left; `BlastRadius`→`PriorPrs`→`ReviewFocus` stacked right) will NOT bottom-align its two columns with `alignItems:"start"` (its original value): each column sizes to its own content, so when the Intent column is taller the Blast card floats short (user report: "Intent and Blast Radius have different heights"), and when Prior PRs is expanded the right column grows past a short Intent card. The design requires the columns to always be equal height / bottom-aligned regardless of Prior-PRs expand state. The fix is `alignItems:"stretch"` on the grid PLUS flex-growing the SINGLE-CARD column ONLY: (1) grid `alignItems:"stretch"` so both grid cells share the taller column's height, AND (2) flex-grow the card in the column that holds exactly ONE card — the LEFT `IntentCard` (wrap its `<section>` in `{display:"flex",flexDirection:"column",flex:1,minHeight:0}` and give `s.card` `flex:1`; add the section style to the loading branch too or it shifts on data load). CRITICAL — do NOT flex-grow a card in the RIGHT column (`BlastRadius`): that column stacks multiple cards, and growing the top one into the slack produces a GIANT EMPTY BORDERED card whenever Intent is much taller than a sparse Blast (e.g. degraded index / 0 callers — a real, reported "buggy view"). A single stretching card only looks right when it's content-rich (Intent) — a mostly-empty stretched card reads as broken. So: left Intent flex-grows; right Blast/Prior/Review stay NATURAL height and the right cell's leftover space is plain background whitespace (no border) below the last card. Net behavior: Prior PRs expanded (right taller) → Intent grows with internal empty space at its card bottom (matches design); collapsed (Intent taller) → right column keeps natural-size cards top-aligned with clean whitespace below (NOT a stretched Blast). `alignItems:"start"` is wrong (right column overflows past a short Intent when Prior expands = the original misalignment). No circular grid-sizing because `flex-grow` distributes space WITHIN a stretched flex item and doesn't change its content-based grid-track contribution. Files: `OverviewTab/styles.ts`, `IntentCard/{styles.ts,IntentCard.tsx}`. (Complements the "height-stable skeletons, not `return null`" note below.)
- The BlastRadius "Index degraded — results may be incomplete" badge (`s.degradedBadge`) must be `display:"flex"`, NOT `"inline-flex"`, to render full-width. It originally sat OUTSIDE the card as a direct child of a flex-column `<section>`, where the default `align-items:stretch` made even an `inline-flex` element span full width. When it was moved INSIDE the card (`s.card`, a plain `display:block` container) to make the notice "part of the card", that stretch no longer applied and an `inline-flex` badge silently collapsed to text-width. Fix: `display:"flex"` (block-level flex fills the container width; left-aligned content keeps icon+text at the start). General trap: an element that relied on a flex parent's stretch for its width will shrink when relocated into a non-flex block context — set `display:"flex"` on it explicitly.

- The Multi-Agent Review Columns-view finding card (`multi-agent/_components/ColumnsView/AgentFindingMini.tsx`, styles.ts `findingLoc`) rendered `finding.file:start_line` as a plain mono div with NO `overflow`/`whiteSpace`/`textOverflow` styling, which looked "trimmed" only INTERMITTENTLY in the narrow 220px column. Root cause is CSS's default line-breaking (UAX #14): "-" is a break opportunity, "/" is NOT. A path whose segments happen to contain hyphens (e.g. `event-types-listing-view.tsx`) wraps onto multiple lines fine; a path with no hyphens (e.g. `packages/trpc/server/routers/viewer/eventTypes/handleChildrenEventTypes.ts`) has zero break opportunities, renders as one unbroken line, overflows, and gets hard-clipped by the ancestor `.column` div's `overflow:hidden` with NO ellipsis — cut off mid-word (e.g. "…routers/v"). This is why the bug read as "sometimes not trimmed correctly" rather than a constant, obviously-broken layout. Fix: give the path div `overflow:"hidden", textOverflow:"ellipsis", whiteSpace:"nowrap"` — the same triad already used for finding TITLE truncation in `client/src/components/FindingsCounts/FindingsPopover.tsx:128-131` — plus a `title={`${file}:${line}`}` attribute so the full path stays reachable on hover. No `truncatePath`/`shortenPath` helper exists anywhere in `client/src` (only `lineLabel()` helpers format the `:line` suffix) — this ellipsis-triad-on-a-fixed-width-container is the only established convention for path/text truncation; reach for it before inventing JS-side path-shortening logic.

## Codebase Patterns

- A `styles.ts` that exports `const s = { … } as const` (with `CSSProperties` values) cannot include a `Record<SomeEnum, { c: string; bg: string }>` map inside the same object — TypeScript rejects it because the Record type is not assignable to `CSSProperties`. Export the palette map as a separate named constant (`export const severityColor: Record<RiskSeverity, { c: string; bg: string }> = { … }`) and import both `{ s, severityColor }` in the component. Pattern established in `RisksCard/styles.ts`; mirror it for any future card that needs a per-severity/per-state color lookup alongside inline styles.

- The PR-list FINDINGS popover re-derives its finding list CLIENT-SIDE in `components/FindingsCounts/aggregate.ts` (`aggregateLatestPerAgent`), which is a SECOND copy of the rule the server uses to compute the `findings_counts` badge in `server/src/modules/pulls/routes.ts` (latest review per `(pr, agentId)`, `kind === 'review'` only, dismissed findings INCLUDED, sorted severity→confidence). The badge total and the popover list length only match because both apply the identical rule — change one and you MUST change the other, or the header "N findings" disagrees with the rows shown. The popover deliberately reuses the existing `GET /pulls/:id/reviews` endpoint (no new endpoint / contract field) and fetches lazily on hover, which is why the client must re-aggregate rather than read a server-prepared list.
- A hover popover/dropdown anchored inside the PR-list table will be CLIPPED: `styles.ts` `tableCard` sets `overflow: hidden` (needed for its rounded corners), so an absolutely-positioned child is cut at the card edge. Render the floating layer via `createPortal(node, document.body)` with `position: fixed` coords from the trigger's `getBoundingClientRect()` instead (see `PRRow/FindingsCell.tsx`). Keep it open across the trigger→popover gap with a short close-delay timer that both elements' `onMouseEnter` cancel, and `stopPropagation` on the popover so clicks don't fire the row's navigate-onClick.
- A PR-list row component (`PRRow`) now fetches lazily on hover via `usePrReviews` (TanStack Query). Any test that renders `PRRow` must wrap it in a `QueryClientProvider` or `useQuery` throws "No QueryClient set" — even though the query is `enabled:false` until hovered (so no fetch mock is needed for the closed/default state). Mirror `agents/_components/AgentCard/AgentCard.test.tsx`.
- The PR-list table is a POSITIONAL CSS grid, not a column-config map. Adding/removing a column means editing THREE places in sync in `src/app/repos/[repoId]/pulls/`: `constants.ts` `GRID` (add a track width — the string order = visual order), `constants.ts` `COLUMN_KEYS` (the header auto-renders from this array in `page.tsx` via `t("list.columns.<key>")`), and `_components/PRRow/PRRow.tsx` (insert the cell `<div>` at the MATCHING index — cells are emitted in source order with no key binding). Miss one and headers silently misalign from cells. Reuse `SEV`/`Icon` from `@devdigest/ui` (re-exported via the primitives barrel) for severity color+icon instead of redefining maps.
- "Findings by severity" is TWO separate features that look identical but live at different altitudes — don't conflate them (a reviewer flagged exactly this). (a) PR-LIST badge: `components/FindingsCounts/*` + `PRRow/FindingsCell.tsx` — read-only counters + hover card over the cross-agent aggregate (`aggregateLatestPerAgent`). (b) PR-DETAIL inline filter: `pulls/[number]/_components/FindingsPanel/` — clickable severity chips in the panel toolbar that filter that ONE review run's list in place. The detail filter is the asked-for feature; the list badge is a bonus. Both reuse the same `SEV` (color/bg/icon/label) + `Icon` tokens from `@devdigest/ui` — never redefine severity maps. Detail-filter wiring: `sevFilter` state in `FindingsPanel`; `visibleFindings(findings, hideLow, sevFilter)` (in `helpers.ts`) applies hide-low THEN the severity filter THEN the severity sort; `countBySeverity()` (pure) tallies chips over the POST-hide-low set so a chip's number always equals the rows shown when it's selected; reset `focusIdx` to 0 on filter change so j/k nav never points past the shrunken list; zero-count chips render `disabled`. Chip click toggles (click active chip → clear). Tested with `fireEvent.click` (NO `@testing-library/user-event` in this package) querying chips by their i18n `aria-label` `panel.filterSeverity` ("Show only {severity} ({count})").
- The page content area gets NO padding from the shell: `@devdigest/ui` `AppFrame` renders `<main style={{ flex:1, minHeight:0, overflow:"auto" }}>{children}</main>` with zero gutter, and there is no global `main` CSS rule. So a page whose top-level child has no padding renders its heading FLUSH against the sidebar divider (visible bug). Every page must supply its own horizontal gutter — the established value is `padding: "24px 32px …"` (see the PR list's `pulls/styles.ts` `pageHeader` `"24px 32px 10px"` + table `margin: "14px 32px …"`, and `SkillsWorkspace` panels at `padding: 28`). When scaffolding a new page under `AppShell`, wrap the content in a `padding: "24px 32px 44px"` container; the breadcrumb bar (rendered separately by AppFrame's header) is already padded, so an unpadded body looks misaligned next to it.
- A scroll-spy / active-section tracker (e.g. the Onboarding Tour's "ON THIS PAGE" `AnchorNav`) must NOT listen for `window` scroll — because AppFrame's `<main overflow:auto>` is the scroll container, `window` itself NEVER scrolls, so a plain `window.addEventListener("scroll", …)` never fires and the rail only ever updates on click (the reported "click Generation cost, scroll up, rail stays stuck" bug). Fix in `tour/_components/TourWorkspace/useScrollSpy.ts`: register in the CAPTURE phase — `window.addEventListener("scroll", fn, true)` — which receives a descendant scroller's scroll events even though `scroll` doesn't bubble; throttle with `requestAnimationFrame`; pick the active anchor as the LAST section whose `getBoundingClientRect().top <= ~160px` (a "reading line" heuristic that also handles the bottom-most/short last card). The hook must be called UNCONDITIONALLY before `TourWorkspace`'s early returns (rules of hooks) and gated by an `enabled` flag (`data.availability === "ready"`) so it no-ops until the section elements mount.

## Codebase Patterns

- The vendored `@devdigest/shared` exports TWO types named `IndexStatus` with COMPLETELY DIFFERENT shapes:
  (1) the shared contract `IndexStatus` (from `contracts/platform.ts`) is a Zod object schema describing
  real-time progress `{ status: 'idle'|'cloning'|..., pct, message, chunks_indexed }`. (2) the
  server's `repo-intel/types.ts` also exports `IndexStatus` as a string union
  `'full'|'partial'|'degraded'|'failed'` representing the FINAL index health state. When the client
  needs to mirror the blast response's `index.status` field (the health string), it must NOT reuse the
  shared-contract `IndexStatus` — define a local alias (`BlastIndexStatus`) instead. This clash is
  invisible until you read both files; grep alone will mislead you.

- Deep-linking from the PR-list findings tooltip to a specific finding's card on the detail page: build `/repos/{repoId}/pulls/{number}?tab=findings#finding-{id}` (the card renders `data-finding-id={f.id}`). Read the finding id from `window.location.hash` in a mount `useEffect` — `useSearchParams()` does NOT include the hash, so a `?finding=` query would be simpler but the existing `?tab=` plumbing already drives the tab. Threading: `page.tsx` → `FindingsTab` → `ReviewRunAccordion` → `FindingsPanel`. CRUCIAL: a finding lives inside a COLLAPSIBLE `ReviewRunAccordion`, and `FindingsPanel` (hence the `FindingCard`) only mounts when the accordion is OPEN — so `ReviewRunAccordion` must detect `review.findings.some(f => f.id === focusFindingId)` and `setOpen(true)` BEFORE `FindingsPanel`'s scroll-into-view effect can find the element. Inside `FindingsPanel`, the focus effect keys on `[focusFindingId, shown]` (not `[]`) so it re-runs once the list renders. When a row links in-app, keep the `file:line` GitHub `<a>` a SIBLING of the title `<Link>`, never nested — nested `<a>`/`<Link>` is invalid HTML (the title row is the in-app target; the file ref opens GitHub in a new tab).
- The detail-page finding GitHub link needs the PR NUMBER, not the head sha (the old `githubBlobUrl` used `headSha`). When switching `FindingCard`/`FindingsPanel`/`FindingsTab`/`ReviewRunAccordion` to `githubPrFileUrl`, replace the whole `headSha` prop chain with `prNumber` (sourced from `pr.number` in `page.tsx`) rather than passing both — `headSha` was link-only plumbing.
- Before building "new" CRUD UI in `client/`, grep `src/lib/hooks/` for an existing mutation hook — the data plumbing may already exist fully but be UNWIRED (no callers). Concrete case: adding skill-deletion to the Skills page was a UI-only task because `useDeleteSkill()` (`src/lib/hooks/skills.ts`, calls `api.del('/skills/:id')`, invalidates `["skills"]`, removes `["skill", id]`) AND the server `DELETE /skills/:id` + FK cascades already existed and were tested — only the trigger was missing. The hooks file ships create/update/**delete**/import + agent-link hooks together even when a screen wires only some of them, so `grep -rn useDeleteSkill src/` returning ONLY the definition line is the tell that the feature is "wire it up", not "build it". Verify before scoping: a hook with zero usages ≠ a missing feature.
- A settings sub-value that lives as a passthrough key on the single `Settings` object (e.g. `root_folders`, added by SPEC-01 T4 with no dedicated endpoint) should get its own hook file (`hooks/settings.ts`) that reads the SAME `["settings"]` query key as `useSettings`/`useUpdateSettings` (`hooks/core.ts`), using TanStack Query's `select` to project out just that field (with a client-side default when absent) — do NOT invent a separate query key. A separate key (e.g. `["root-folders"]`) would issue its own `GET /settings` fetch and require its own invalidation on every mutation that touches the shared row, silently drifting out of sync with the rest of the Settings panel. The paired mutation hook still does its own `PUT /settings` with just the one key in the body (the server upserts key-by-key) and writes the full response back via `qc.setQueryData(["settings"], data)` so both hooks re-derive from one source of truth.
- A master/detail screen whose selection lives in the URL (`?doc=` query, not a route segment) can be exercised in RTL by mocking `next/navigation`'s `useSearchParams` to read a module-level mutable string that the mocked `useRouter().push` writes to when given a `?doc=...` URL, THEN calling RTL's `rerender(...)` on the same tree after the click — `rerender` re-invokes the (non-memoized) function component body, which re-reads the updated mock, without needing a full router-mock library or `useSyncExternalStore` plumbing in the mock itself. See `ContextWorkspace.test.tsx` "selecting a doc shows its preview".
- When a page's thin `page.tsx` is restricted to owning ONLY that file (a plan's "owned paths" excludes the feature's `_components/<Workspace>` folder), you cannot add UI to that workspace's existing `<AppShell crumb=…>` header/breadcrumb without violating the boundary — even a "add a link to X" task. The workaround used for the Conventions→Context cross-link (SPEC-01 T11): render an extra sibling `<div>` bar BEFORE `<ConventionsWorkspace/>` directly in `conventions/page.tsx`. Because `ConventionsWorkspace` renders its OWN `<AppShell>` (a `display:flex, minHeight:100vh` row = sidebar + content), a normal (non-fixed) sibling bar placed before it in the DOM renders as a full-width strip ABOVE the entire app shell (sidebar included) rather than inside the page content area — that's an accepted trade off of not touching the owned workspace file, not a bug.
- The vitest CLI glob patterns some plans hand you (e.g. `vitest run "src/app/repos/[repoId]/context/_components/**/*.test.tsx"`) do NOT match on this codebase's App Router bracket-folder names: `[repoId]` is valid glob character-class syntax (matches exactly ONE char from the set `{r,e,p,o,I,d}`), so tinyglobby/fast-glob interprets that path segment as a 1-char match against an 8-char literal directory name and reports "No test files found, exiting with code 1" — this reproduces identically with single quotes, double quotes, or backslash-escaped brackets (`\[repoId\]`; the shell strips the backslash before the glob engine ever sees it, so escaping doesn't help either). It is NOT a sandboxing or environment issue. Two reliable ways to actually run the target test: (1) pass the LITERAL file path with no `**` wildcard (e.g. `vitest run "src/app/repos/[repoId]/context/_components/ContextWorkspace/ContextWorkspace.test.tsx"`) — vitest checks for an exact existing file before treating the argument as a glob, so a literal path with brackets resolves fine; (2) just run the bare `vitest run` (picks up the config's own `include: src/**/*.test.{ts,tsx}`, which is a different, already-working glob) and grep the file's name out of the full output.
- Running `node_modules/.bin/vitest run` for the WHOLE client suite (~200+ tests) inside this environment's default sandbox occasionally dies with an unexplained `exit code 1` and a truncated log (no `Test Files … passed` summary line at all, just stops mid-suite) when multiple vitest invocations are in flight concurrently (e.g. several `run_in_background` Bash calls stacked while waiting on notifications) — this is sandbox resource contention (confirmed via `ps aux | grep vitest` showing many stray `node (vitest N)` worker processes from overlapping runs), not a real test failure. Kill strays (`pkill -9 -f vitest`) and either run the full suite with `dangerouslyDisableSandbox: true` or avoid launching more than one full-suite `vitest run` at a time; a clean single run of the full suite passes (34 files / 205+ tests) without the flag once nothing else is contending.

- **A shared presentational component consumed by two features with DIFFERENT `next-intl` namespaces (e.g. Agent vs Skill Context tabs, both under `context.*` but namespaces `"agents"` vs `"skills"`) should accept `t: ReturnType<typeof useTranslations>` as a prop, NOT a namespace string.** This project has no `IntlMessages` global type augmentation, so `useTranslations(ns)` is NOT namespace-typed — `ReturnType<typeof useTranslations>` is the same type regardless of which namespace string was passed, so a parent's `useTranslations("agents")` and `useTranslations("skills")` are interchangeable at the type level as long as both namespaces expose the same key paths (verify by diffing `messages/en/<ns>.json` — they don't need identical VALUES, just identical KEY SHAPE). Precedent for this exact pattern already existed before `DocumentAttachmentPicker`: `TraceBody.tsx`'s internal `OriginBadge({ t }: { t: ReturnType<typeof useTranslations> })`. Do NOT invent a `labels` object or pass the namespace string itself into the shared component — that would require it to call `useTranslations` internally and lose the parent's actual scoped keys.
- **When extracting a shared component out of two near-duplicate feature components (dedup refactor), keep both original `*.test.tsx` files unchanged** if they already render the FULL feature end-to-end (mocking only the data hooks, not the JSX) — they continue to exercise the extracted shared component's rendering logic transitively with zero test-file changes, so there's no coverage gap to fill with a separate `SharedComponent.test.tsx`. Concrete case: `AgentEditor/ContextTab.test.tsx` and `SkillEditor/ContextTab.test.tsx` (SPEC-01 follow-up dedup into `components/DocumentAttachmentPicker`) needed NO test changes after the refactor — both wrappers still render `<DocumentAttachmentPicker>` under the hood, and the existing tests assert on rendered text/aria-labels/mutation calls, not on which component tree produced them.

- Testing a `useQuery({ refetchInterval: (query) => ... })` self-clearing poll (the `usePrRuns`/`useMultiAgentRun` pattern) with vitest fake timers is UNRELIABLE and should be avoided — don't try to simulate the actual 4s wall-clock tick. Two failure modes hit in the same debugging session: (1) `vi.useFakeTimers({ shouldAdvanceTime: true })` lets its background real-time auto-tick drift the fake clock ahead of explicit `vi.advanceTimersByTimeAsync(N)` calls, so a manual advance can under- or over-shoot exactly which scheduled refetch fires; (2) even with plain `vi.useFakeTimers()` (no auto-advance) wrapped in `act(...)`, the query observer's cache update from a timer-driven refetch consistently landed ONE tick behind what `result.current.data` showed immediately after `await act(() => vi.advanceTimersByTimeAsync(4000))` returned — `apiGet` had provably already been CALLED (call count matched), but the resolved value hadn't yet propagated into the hook's render snapshot, and neither an extra `advanceTimersByTimeAsync(0)` flush nor a `waitFor(...)` (which falls back to a real-time `checkRealTimersCallback` poll under detected fake timers, capped at ~1000 real ms) reliably caught up — the mismatch reproduced consistently across repeated runs, not just occasionally. The fix that IS reliable: don't drive the timer at all — render the hook once (real timers, normal `waitFor(isSuccess)`), then reach into the live `QueryClient` for the actual wired predicate: `qc.getQueryCache().find({ queryKey })!.observers[0].options.refetchInterval` (typed `(query: { state: { data?: T } }) => number | false`), and call THAT function directly with synthetic `{ state: { data: fixture(...) } }` snapshots. This exercises the exact function TanStack Query calls to decide whether to keep polling — sourced from the real hook via a real `QueryClient`, not a hand-copied reimplementation — while being fully deterministic (no timer simulation at all). See `client/src/lib/hooks/multi-agent.test.tsx`'s `useMultiAgentRun` "polls every 4s … stops" test.

## Recurring Errors & Fixes

- A case-only file rename (e.g. `atoms.tsx` → `Atoms.tsx`) silently UN-records itself after a `git reset` on macOS's case-insensitive filesystem. Sequence that bites: `git mv atoms.tsx Atoms.tsx` stages `R atoms.tsx -> Atoms.tsx`; a later `git reset` (to do per-file staging) unstages it; git then compares the tracked lowercase name against the on-disk `Atoms.tsx`, sees them as the SAME path (case-insensitive), and reports the dir CLEAN — so a normal `git add`/commit captures the import edits but NOT the rename. `git ls-files` still shows `atoms.tsx` while `ls` shows `Atoms.tsx`. This passes local typecheck/tests/build (case-insensitive FS resolves either way) but BREAKS on case-sensitive CI/Linux where `import … from "./Atoms"` can't find `atoms.tsx`. Fix: force the rename explicitly with `git mv -f atoms.tsx Atoms.tsx` (then commit/`--amend`); verify with `git show --stat` showing `{atoms.tsx => Atoms.tsx}`. Avoid the `git reset` re-stage dance for case renames, or re-apply `git mv -f` afterward.
- A cell component rendered by `PRRow` (e.g. `FindingsCell`) that calls `useParams()` crashes `PRRow.test.tsx` — that test does `vi.mock("next/navigation", () => ({ useRouter: ... }))` and provides ONLY `useRouter`, so `useParams` is `undefined` → throws inside the render. Don't reach for `useParams` in PR-row cells: `PRRow` already RECEIVES `repoId` as a prop, so thread it down (`<FindingsCell pr={pr} repoId={repoId} />`). (`useActiveRepo()` is safe in that test — its context has a default value — so reading `activeRepo.full_name` from it does not need a provider.)
- RTL `getByText(/whole string/)` fails ("Unable to find an element") when JSX interpolates siblings: `{a} tok · {formatUsd(b)}` renders THREE text nodes ("9,731", " tok · ", "$0.012"), and `getByText` matches per-node by default. Fix in the component (not the test) by collapsing to one text node with a template literal: `{`${a} tok · ${formatUsd(b)}`}`. Done for the run cost line in `RunHistory.tsx`.
- `getByText("—")` throws "Found multiple elements" once a row can render the em-dash placeholder in MORE than one cell (e.g. `PRRow` shows `—` for both null `cost_usd` and null `findings_counts`). Adding a new dash-able cell breaks pre-existing single-dash assertions. Fix: in the shared test fixture builder give the new field a NON-dash default (e.g. `findings_counts: {critical:2,warning:2,suggestion:1}`), and in each dash-specific test set the OTHER cells non-null so exactly one `—` remains. Prefer `getByLabelText(...)` for the new cell (e.g. the `FindingsCounts` group's aria-label) — it's unique regardless of how many dashes render.
- React (Next 15) logs a runtime console error — "Updating a style property during rerender (`borderColor`) when a conflicting property is set (`borderLeftColor`) can lead to styling bugs" — whenever a base `style` object sets the `border` SHORTHAND and a spread inline `style` overrides a longhand like `borderColor` on the SAME element (React expands `border` into `borderLeftColor` etc., so the two collide on rerender). Hit when a `FindingsPanel/SeverityFilter` chip merged `{...s.chip, borderColor: active ? meta.c : ...}`. Fix: in the base style object, never use the `border` shorthand if any per-render override touches the border — split it into `borderWidth`/`borderStyle`/`borderColor` longhands so only longhands are ever set. Same rule applies to `background`/`margin`/`padding` shorthands paired with longhand overrides.
- FOLLOW-UP / GOTCHA on the above: `borderColor` is ITSELF a shorthand for the four per-side colors, so removing the literal `border` shorthand is NOT enough — an element that sets BOTH `borderColor` (all sides) AND `borderLeftColor` (a per-side accent) STILL trips the identical "(`borderColor`) … conflicting property … (`borderLeftColor`)" warning the moment `borderColor` changes between renders. Real case: `FindingCard/styles.ts` `s.card()` draws a severity-colored left accent (`borderLeftColor: sevColor`, `borderLeftWidth: 3`) plus a focus ring (`borderColor: focused ? sevColor : 'var(--border)'`); toggling `focused` (e.g. j/k nav, or `FindingsPanel` resetting `focusIdx` on a severity-filter click) updated `borderColor` and fired the error in the console. Latent bug, just easy to trigger via the filter. Fix: drop `borderColor` entirely and set the three NON-left sides explicitly — `borderTopColor`/`borderRightColor`/`borderBottomColor` (= the focus color) alongside the separate `borderLeftColor`. Rule of thumb: if ANY side color is set individually, set ALL four individually; never mix `borderColor` with a `border<Side>Color`.
- `"GitCompare"` is **not** in the `@devdigest/ui` icon registry — referencing it silently resolves to `undefined` and renders nothing. For Smart Diff / risk-ordered-view concepts, use `"Layers"`. Before naming an icon in `PrDetailHeader` or any other component that reads from the UI kit's icon map, verify it exists in `client/src/vendor/ui/` (grep for the name). `"Code"` is already taken by the Files-changed tab.
- The diff-viewer barrel (`client/src/components/diff-viewer/index.ts`) originally exported only `DiffViewer` — `parsePatch` and `Line` were NOT exported. Any component that needs `parsePatch` (e.g. `SmartDiffViewer`) must either import from the deep path (`@/components/diff-viewer/helpers`) or add an additive re-export to the barrel first. The additive re-export is preferred (avoids deep internal imports) and was done for Smart Diff: `export { parsePatch, Line } from "./helpers"` added to `index.ts`.
- `vi.useFakeTimers()` + RTL `waitFor` is a deadlock: `waitFor` polls via `setTimeout`/`setInterval` — both faked — so it never advances and the test times out at 5 000ms. - To navigate programmatically from the SmartDiffViewer (Files-changed tab) to a specific finding in the Agent runs tab, use the EXISTING `focusFindingId` React state in `page.tsx` — do NOT introduce a new state or URL param. - In `SmartDiffViewer`'s `DiffLine`, `lineNo = ln.newNo ?? ln.oldNo`. A del/add pair at the same logical position in a unified diff (e.g., `-  return x;\n+  return x + 1;`) both resolve to `lineNo = 12` because the del has `oldNo=12, newNo=null` → `lineNo=12`, and the add has `newNo=12, oldNo=null` → `lineNo=12`. Consequence: when a `FindingAnnotation` cites line 12, **both** the del and add rows will each render a severity pill. This is correct from a user POV (both rows are highlighted), but RTL tests asserting `getByText(/warning/i)` will throw "multiple elements found" — use `getAllByText(/^warning$/i)` and assert `length >= 1` instead.

- `SmartDiffViewer` has TWO visually similar badge-click behaviors that navigate completely differently — don't conflate them. (A) **File-level badge** ("N findings" chip on the card header): `handleFindingsBadgeClick` (`SmartDiffViewer.tsx:200–211`) expands the file accordion if collapsed, then waits one tick (`setTimeout(0)`) before calling `document.getElementById(lineDomId(path, firstAnnotation.line))?.scrollIntoView({ block: "center" })`. The `setTimeout(0)` is required because the accordion expansion is a React state update that needs a render cycle before the line DOM nodes exist — without it, `getElementById` finds nothing. - RTL `getByText` (and `getAllByText`) uses `getNodeText(node)` which reads **only the node's DIRECT text nodes** (children where `childNode.nodeType === TEXT_NODE`) — NOT the concatenated text of all descendants. This means a `<div>` that holds only child `<span>` elements has `getNodeText` = "" even if those spans each contain text. Practical upshot: splitting a stat row into four `<span>` elements (icon + template-literal text per stat) does NOT create multiple-match collisions when you query `getByText(/1 symbols/)` — the parent flex container returns "" and the sibling span with "2 callers" doesn't match "/1 symbols/".
