## What Doesn't Work

- The BlastRadius panel (`pulls/[number]/_components/BlastRadius/BlastRadius.tsx`) originally computed `isEmpty = !data || totals.symbols === 0 || !hasCallers`, where `hasCallers = data.symbols.some(g => g.callers.length > 0)`. The `|| !hasCallers` term COLLAPSES THE WHOLE PANEL to the one-line `noDownstream` message whenever NO changed symbol has a resolved downstream caller ‚Äî so a PR with 53 changed symbols but 0 callers renders as just text, with every symbol (and the Tree/Graph tree) hidden. This came straight from the plan (`docs/plans/blast-radius.md:218` "when `totals.symbols === 0` or no callers ‚Üí explicit empty state"), which contradicts the plan's OWN acceptance criterion ("empty state when `totals.symbols === 0`", lines 260/296). "0 callers" is common and expected: `getResolvedCallers` (`server/.../repo-intel/repository.ts`) is precision-over-recall ‚Äî it only counts `references` rows whose resolved `declFile` ‚àà changed files (NULL `declFile` dropped), so re-exported/barrel symbols or unresolved refs yield 0 callers even on a full index (no degraded badge shown). Fix: gate empty ONLY on symbols (`isEmpty = !data || totals.symbols === 0`); when `totals.symbols > 0 && totals.callers === 0`, still render the symbol tree and show the `noDownstream` line as a NOTE above it (`s.noCallersNote`), and add a distinct `blast.empty` ("No impacted symbols found for this PR.") for the true-zero case so the count-0 `noDownstream` string isn't shown. `SymbolRow` already renders fine with `callers: []` (the caller `<ul>` is conditional), and `BlastGraph` already guards `hasContent` ‚Üí its own empty message. Regression test: a fixture with symbols but `callers: []` must assert the symbol NAMES render (not the empty state).
- The Skill ConfigTab (`skills/[id]/_components/SkillEditor/_components/ConfigTab/ConfigTab.tsx`) must PATCH only the fields that actually changed ‚Äî sending the whole form on every save DEFEATS the server's body-only versioning. The server versions a skill by its BODY only (`server/src/modules/skills/{helpers.ts:isBodyChange, repository.ts:update}`; `skill_versions` stores body snapshots): a DRAFT (v0) commits to v1 on the first save that *includes a body*, and a committed skill bumps + snapshots ONLY on a real body change ‚Äî name/description/type/enabled edits never version. The original ConfigTab always sent `{name, description, type, body, enabled}`, so a metadata-only save on a draft resent the unchanged scaffold body and prematurely committed v1 (burning a version on the scaffold the server explicitly tries to avoid). This is also why "edit something, save ‚Üí always v1" looks like a bug: metadata edits legitimately don't version, and the always-full-patch hid that. Fix: build `patch` from per-field diffs vs the `skill` prop (`if (name !== skill.name) patch.name = name; ‚Ä¶`) and send only changed keys; gate the Save button with an `isDirty` flag (`disabled={update.isPending || !isDirty || !name || !body}`) so it's off until something changes. Test with `fireEvent.change` (no user-event in this package) then assert `mutate` got `{ patch: { <onlyChangedField> } }`; a no-change render must leave Save `toBeDisabled()`.
- Linking a finding's `file:line` to GitHub via `githubBlobUrl` (`/{repo}/blob/{sha}/{file}#L{line}`) 404s in two real cases: (a) FORK PRs ‚Äî `pr.head.sha` lives on the contributor's fork, not the base repo, so `base-repo/blob/{forkSha}` doesn't exist; (b) the demo SEED ‚Äî `seed.ts` uses a fabricated repo (`acme/payments-api`) + fake `head_sha` (`a1b2c3d4e5f6`), so EVERY github.com link 404s regardless of format (a user reported "always 404" here; the format was fine, the data isn't real). Fix: link into the PR's Files-changed diff with `githubPrFileUrl` (`/{repo}/pull/{number}/files#diff-<sha256(path)>R{line}` in `lib/github-urls.ts`) ‚Äî the Files view is owned by the BASE repo, so it resolves for fork PRs. The `#diff-` hash is the SHA-256 hex of the repo-relative path, computed async via Web Crypto in the `usePathShas` hook (`lib/hooks/use-path-shas.ts`); render the bare `/pull/{n}/files` URL immediately and upgrade once the hash resolves (and when `crypto.subtle` is missing ‚Äî jsdom/insecure context ‚Äî keep the bare URL). `R{line}` targets the added/right side; it only anchors precisely if the line is in the diff, but the file still loads otherwise. To verify a real (non-404) link you MUST import a real public repo ‚Äî the seed can't resolve.
- The vendored `Checkbox` (`src/vendor/ui/kit/Checkbox.tsx`, a `<button>` inside a `<label>`) fires `onChange` **TWICE** per physical click (the label re-dispatches activation to its labeled control). Beyond the server-side concurrency this causes (`server/INSIGHTS.md` advisory-lock note), it also produces a LOGICAL toggle add-back in `AgentEditor/_components/SkillsTab/SkillsTab.tsx`: its `toggle(id, on)` trusts the `on` (`!checked`) the checkbox passes and closes over `attached`; when React flushes a re-render BETWEEN the two synthetic clicks, the second click reads the now-flipped `checked` and computes the OPPOSITE intent ‚Üí it re-adds the skill you just removed, so disabling a skill silently leaves it enabled. It's a RACE: small lists (1‚Äì2 linked skills) render fast enough to flush between the two clicks ‚Üí reproduces ~1-in-5; larger lists almost never do ‚Üí "works 80% of the time". The two `POST /agents/:id/skills` carry DIFFERENT `skill_ids` (`[‚Ä¶without id]` then `[‚Ä¶with id]`), so the server advisory lock can't fix it ‚Äî last writer wins. FIX (vendor is "do not touch", so client-side): a per-skill-id in-flight guard `const toggling = React.useRef<Set<string>>(new Set())` ‚Äî `toggle` returns early if `toggling.current.has(id)` (drops the spurious synchronous second fire), else adds the id and clears it via the mutation's `onSettled` (`setSkills.mutate(ids, { onSettled })`). The second fire is dispatched synchronously within the same click gesture, always inside the network round-trip window, so it's reliably ignored; `onDrop`/reorder is keyed differently and passes no `onSettled` so it's unaffected. Regression test (`SkillsTab.test.tsx`): `fireEvent.click` the same checkbox TWICE back-to-back, assert `setMutate` called ONCE with `[]` (verified it fails ‚Äî 2 calls ‚Äî without the guard). A single `await`-each / single-click test proves nothing.
- A `position: fixed` portal popover positioned from a ONE-TIME `getBoundingClientRect()` (e.g. `FindingsHoverCard`) DETACHES on scroll: `fixed` pins it to the viewport, so when the page/timeline scrolls the trigger moves but the card stays floating over unrelated content. Must dismiss (or reposition) on `scroll`+`resize` while open. Two subtleties: (a) register the `scroll` listener in the CAPTURE phase (`window.addEventListener('scroll', fn, true)`) ‚Äî `scroll` does NOT bubble, and the content may live in a NESTED `overflow:auto` scroller a non-capture window listener would miss; (b) IGNORE scrolls originating inside the popover's own `overflow:auto` list via `popoverRef.current?.contains(e.target)`, or scrolling a long findings list self-dismisses the card. Regression test: open via `mouseEnter`, `fireEvent.scroll(window)`, assert it's gone (`RunHistory.test.tsx`).
- A master/detail page whose LIST is its own `overflow:auto` scroll container (e.g. the Skills sidebar in `skills/_components/SkillsWorkspace/SkillsWorkspace.tsx`) JUMPS that list back to the top whenever you select a row, because Next.js App Router's `router.push`/`router.replace` default to `scroll: true`, which scrolls the nearest scroll container ‚Äî NOT just `window` ‚Äî to the top on every navigation. Symptom a user will report: "scroll down, click the last skill, the list snaps back to the top." Fix: pass `{ scroll: false }` to EVERY intra master/detail navigation ‚Äî the row-select push (`router.push(`/skills/${id}?tab=${tab}`, { scroll: false })`) AND the tab-change `router.replace`s in BOTH `skills/page.tsx` and `skills/[id]/page.tsx` (a tab switch shouldn't move the list either). There's no visible window scrollbar to hint at this ‚Äî the layout is `height: calc(100vh - 52px)` with inner scrollers, so the reset only ever hits the inner container.
- `BlastGraph` (`BlastRadius/BlastGraph.tsx`) early-returns its "No downstream callers to graph." message when `callers.length === 0 && rightNodes.length === 0`, so a PR with N changed symbols but 0 resolved callers/endpoints renders a BLANK graph and the Tree‚ÜíGraph toggle LOOKS broken (the toggle itself works ‚Äî it's the empty gate). 0 callers is common/expected for route-handler PRs (see the `getResolvedCallers` precision-over-recall note above). Fix: draw the left changed-symbol column whenever `symbols.length > 0`, and only show the empty copy when there are literally no symbols.
- A PR-score gauge already exists ‚Äî do NOT build a new one when asked to add a "score badge" to the Overview/Intent area. `_components/VerdictBanner/VerdictBanner.tsx` renders the verdict pill + summary + `CircularScore` "PR SCORE" and self-guards `score != null`; `CircularScore` is a `vendor/ui` primitive. It's currently mounted ONLY inside `ReviewRunAccordion` (Findings tab), NOT on the Overview ‚Äî but the design places it atop the PR Brief. To wire it: pass the latest review (`reviews[0]` from `usePrReviews`, already fetched in `pulls/[number]/page.tsx`; `verdict`/`summary`/`score` per `vendor/shared/contracts/review-api.ts`) into `OverviewTab` and render `<VerdictBanner>` above `IntentCard`. No new endpoint/contract needed.

## Codebase Patterns

- A `styles.ts` that exports `const s = { ‚Ä¶ } as const` (with `CSSProperties` values) cannot include a `Record<SomeEnum, { c: string; bg: string }>` map inside the same object ‚Äî TypeScript rejects it because the Record type is not assignable to `CSSProperties`. Export the palette map as a separate named constant (`export const severityColor: Record<RiskSeverity, { c: string; bg: string }> = { ‚Ä¶ }`) and import both `{ s, severityColor }` in the component. Pattern established in `RisksCard/styles.ts`; mirror it for any future card that needs a per-severity/per-state color lookup alongside inline styles.

- The PR-list FINDINGS popover re-derives its finding list CLIENT-SIDE in `components/FindingsCounts/aggregate.ts` (`aggregateLatestPerAgent`), which is a SECOND copy of the rule the server uses to compute the `findings_counts` badge in `server/src/modules/pulls/routes.ts` (latest review per `(pr, agentId)`, `kind === 'review'` only, dismissed findings INCLUDED, sorted severity‚Üíconfidence). The badge total and the popover list length only match because both apply the identical rule ‚Äî change one and you MUST change the other, or the header "N findings" disagrees with the rows shown. The popover deliberately reuses the existing `GET /pulls/:id/reviews` endpoint (no new endpoint / contract field) and fetches lazily on hover, which is why the client must re-aggregate rather than read a server-prepared list.
- A hover popover/dropdown anchored inside the PR-list table will be CLIPPED: `styles.ts` `tableCard` sets `overflow: hidden` (needed for its rounded corners), so an absolutely-positioned child is cut at the card edge. Render the floating layer via `createPortal(node, document.body)` with `position: fixed` coords from the trigger's `getBoundingClientRect()` instead (see `PRRow/FindingsCell.tsx`). Keep it open across the trigger‚Üípopover gap with a short close-delay timer that both elements' `onMouseEnter` cancel, and `stopPropagation` on the popover so clicks don't fire the row's navigate-onClick.
- A PR-list row component (`PRRow`) now fetches lazily on hover via `usePrReviews` (TanStack Query). Any test that renders `PRRow` must wrap it in a `QueryClientProvider` or `useQuery` throws "No QueryClient set" ‚Äî even though the query is `enabled:false` until hovered (so no fetch mock is needed for the closed/default state). Mirror `agents/_components/AgentCard/AgentCard.test.tsx`.
- The PR-list table is a POSITIONAL CSS grid, not a column-config map. Adding/removing a column means editing THREE places in sync in `src/app/repos/[repoId]/pulls/`: `constants.ts` `GRID` (add a track width ‚Äî the string order = visual order), `constants.ts` `COLUMN_KEYS` (the header auto-renders from this array in `page.tsx` via `t("list.columns.<key>")`), and `_components/PRRow/PRRow.tsx` (insert the cell `<div>` at the MATCHING index ‚Äî cells are emitted in source order with no key binding). Miss one and headers silently misalign from cells. Reuse `SEV`/`Icon` from `@devdigest/ui` (re-exported via the primitives barrel) for severity color+icon instead of redefining maps.
- "Findings by severity" is TWO separate features that look identical but live at different altitudes ‚Äî don't conflate them (a reviewer flagged exactly this). (a) PR-LIST badge: `components/FindingsCounts/*` + `PRRow/FindingsCell.tsx` ‚Äî read-only counters + hover card over the cross-agent aggregate (`aggregateLatestPerAgent`). (b) PR-DETAIL inline filter: `pulls/[number]/_components/FindingsPanel/` ‚Äî clickable severity chips in the panel toolbar that filter that ONE review run's list in place. The detail filter is the asked-for feature; the list badge is a bonus. Both reuse the same `SEV` (color/bg/icon/label) + `Icon` tokens from `@devdigest/ui` ‚Äî never redefine severity maps. Detail-filter wiring: `sevFilter` state in `FindingsPanel`; `visibleFindings(findings, hideLow, sevFilter)` (in `helpers.ts`) applies hide-low THEN the severity filter THEN the severity sort; `countBySeverity()` (pure) tallies chips over the POST-hide-low set so a chip's number always equals the rows shown when it's selected; reset `focusIdx` to 0 on filter change so j/k nav never points past the shrunken list; zero-count chips render `disabled`. Chip click toggles (click active chip ‚Üí clear). Tested with `fireEvent.click` (NO `@testing-library/user-event` in this package) querying chips by their i18n `aria-label` `panel.filterSeverity` ("Show only {severity} ({count})").
- The page content area gets NO padding from the shell: `@devdigest/ui` `AppFrame` renders `<main style={{ flex:1, minHeight:0, overflow:"auto" }}>{children}</main>` with zero gutter, and there is no global `main` CSS rule. So a page whose top-level child has no padding renders its heading FLUSH against the sidebar divider (visible bug). Every page must supply its own horizontal gutter ‚Äî the established value is `padding: "24px 32px ‚Ä¶"` (see the PR list's `pulls/styles.ts` `pageHeader` `"24px 32px 10px"` + table `margin: "14px 32px ‚Ä¶"`, and `SkillsWorkspace` panels at `padding: 28`). When scaffolding a new page under `AppShell`, wrap the content in a `padding: "24px 32px 44px"` container; the breadcrumb bar (rendered separately by AppFrame's header) is already padded, so an unpadded body looks misaligned next to it.

## Codebase Patterns

- The vendored `@devdigest/shared` exports TWO types named `IndexStatus` with COMPLETELY DIFFERENT shapes:
  (1) the shared contract `IndexStatus` (from `contracts/platform.ts`) is a Zod object schema describing
  real-time progress `{ status: 'idle'|'cloning'|..., pct, message, chunks_indexed }`. (2) the
  server's `repo-intel/types.ts` also exports `IndexStatus` as a string union
  `'full'|'partial'|'degraded'|'failed'` representing the FINAL index health state. When the client
  needs to mirror the blast response's `index.status` field (the health string), it must NOT reuse the
  shared-contract `IndexStatus` ‚Äî define a local alias (`BlastIndexStatus`) instead. This clash is
  invisible until you read both files; grep alone will mislead you.

- Deep-linking from the PR-list findings tooltip to a specific finding's card on the detail page: build `/repos/{repoId}/pulls/{number}?tab=findings#finding-{id}` (the card renders `data-finding-id={f.id}`). Read the finding id from `window.location.hash` in a mount `useEffect` ‚Äî `useSearchParams()` does NOT include the hash, so a `?finding=` query would be simpler but the existing `?tab=` plumbing already drives the tab. Threading: `page.tsx` ‚Üí `FindingsTab` ‚Üí `ReviewRunAccordion` ‚Üí `FindingsPanel`. CRUCIAL: a finding lives inside a COLLAPSIBLE `ReviewRunAccordion`, and `FindingsPanel` (hence the `FindingCard`) only mounts when the accordion is OPEN ‚Äî so `ReviewRunAccordion` must detect `review.findings.some(f => f.id === focusFindingId)` and `setOpen(true)` BEFORE `FindingsPanel`'s scroll-into-view effect can find the element. Inside `FindingsPanel`, the focus effect keys on `[focusFindingId, shown]` (not `[]`) so it re-runs once the list renders. When a row links in-app, keep the `file:line` GitHub `<a>` a SIBLING of the title `<Link>`, never nested ‚Äî nested `<a>`/`<Link>` is invalid HTML (the title row is the in-app target; the file ref opens GitHub in a new tab).
- The detail-page finding GitHub link needs the PR NUMBER, not the head sha (the old `githubBlobUrl` used `headSha`). When switching `FindingCard`/`FindingsPanel`/`FindingsTab`/`ReviewRunAccordion` to `githubPrFileUrl`, replace the whole `headSha` prop chain with `prNumber` (sourced from `pr.number` in `page.tsx`) rather than passing both ‚Äî `headSha` was link-only plumbing.
- Before building "new" CRUD UI in `client/`, grep `src/lib/hooks/` for an existing mutation hook ‚Äî the data plumbing may already exist fully but be UNWIRED (no callers). Concrete case: adding skill-deletion to the Skills page was a UI-only task because `useDeleteSkill()` (`src/lib/hooks/skills.ts`, calls `api.del('/skills/:id')`, invalidates `["skills"]`, removes `["skill", id]`) AND the server `DELETE /skills/:id` + FK cascades already existed and were tested ‚Äî only the trigger was missing. The hooks file ships create/update/**delete**/import + agent-link hooks together even when a screen wires only some of them, so `grep -rn useDeleteSkill src/` returning ONLY the definition line is the tell that the feature is "wire it up", not "build it". Verify before scoping: a hook with zero usages ‚âÝ a missing feature.

## Tool & Library Notes

- Testing a hover popover that `createPortal`s to `document.body` (e.g. `FindingsHoverCard`) in jsdom: (1) `mouseEnter` does NOT bubble, so `fireEvent.mouseEnter` must target the exact element with the handler ‚Äî the wrapper div, reachable as `screen.getByLabelText("‚Ä¶counts aria-label‚Ä¶").parentElement` ‚Äî NOT the inner counters node. (2) The portal still opens even though jsdom's `getBoundingClientRect()` returns all-zeros, because the component stores a truthy `{top:0,left:0}` coords object and renders regardless; assert the revealed content with normal `screen.getByText(...)` since RTL queries default to `document.body` where the portal mounts. No need to stub `getBoundingClientRect`. See `RunHistory.test.tsx` "findings hover popover".
- The vendored `MetricCard` (`src/vendor/ui/charts/MetricCard.tsx`) has only ONE pluggable header slot and it renders a `Sparkline` ‚Äî gated on the `trend` prop. There's no `headerSlot`/`icon` prop, so you CANNOT put a ring gauge / radial dial in a KPI tile through `MetricCard`. The orange dial color is `var(--warn)` (`#f59e0b`), NOT `var(--accent)` (which is BLUE `#3b82f6` in this theme). - There is **no ESLint or Prettier anywhere** ‚Äî not in `client/` and not at the repo root (no `eslint.config.*`/`.eslintrc`/`.prettierrc`, no eslint/prettier deps). So the architecture conventions in `client/CLAUDE.md` (data-access only via `lib/hooks ‚Üí lib/api`, thin pages, `@/*` aliases over relative imports, named exports, feature isolation) are enforced by **convention and review only ‚Äî nothing fails CI if you break them**. Practically: don't assume `pnpm lint` exists (it doesn't); the full client gate is `pnpm typecheck` + `pnpm test` + `pnpm build`. ## Recurring Errors & Fixes

- A case-only file rename (e.g. `atoms.tsx` ‚Üí `Atoms.tsx`) silently UN-records itself after a `git reset` on macOS's case-insensitive filesystem. Sequence that bites: `git mv atoms.tsx Atoms.tsx` stages `R atoms.tsx -> Atoms.tsx`; a later `git reset` (to do per-file staging) unstages it; git then compares the tracked lowercase name against the on-disk `Atoms.tsx`, sees them as the SAME path (case-insensitive), and reports the dir CLEAN ‚Äî so a normal `git add`/commit captures the import edits but NOT the rename. `git ls-files` still shows `atoms.tsx` while `ls` shows `Atoms.tsx`. This passes local typecheck/tests/build (case-insensitive FS resolves either way) but BREAKS on case-sensitive CI/Linux where `import ‚Ä¶ from "./Atoms"` can't find `atoms.tsx`. Fix: force the rename explicitly with `git mv -f atoms.tsx Atoms.tsx` (then commit/`--amend`); verify with `git show --stat` showing `{atoms.tsx => Atoms.tsx}`. Avoid the `git reset` re-stage dance for case renames, or re-apply `git mv -f` afterward.
- A cell component rendered by `PRRow` (e.g. `FindingsCell`) that calls `useParams()` crashes `PRRow.test.tsx` ‚Äî that test does `vi.mock("next/navigation", () => ({ useRouter: ... }))` and provides ONLY `useRouter`, so `useParams` is `undefined` ‚Üí throws inside the render. Don't reach for `useParams` in PR-row cells: `PRRow` already RECEIVES `repoId` as a prop, so thread it down (`<FindingsCell pr={pr} repoId={repoId} />`). (`useActiveRepo()` is safe in that test ‚Äî its context has a default value ‚Äî so reading `activeRepo.full_name` from it does not need a provider.)
- RTL `getByText(/whole string/)` fails ("Unable to find an element") when JSX interpolates siblings: `{a} tok ¬∑ {formatUsd(b)}` renders THREE text nodes ("9,731", " tok ¬∑ ", "$0.012"), and `getByText` matches per-node by default. Fix in the component (not the test) by collapsing to one text node with a template literal: `{`${a} tok ¬∑ ${formatUsd(b)}`}`. Done for the run cost line in `RunHistory.tsx`.
- `getByText("‚Äî")` throws "Found multiple elements" once a row can render the em-dash placeholder in MORE than one cell (e.g. `PRRow` shows `‚Äî` for both null `cost_usd` and null `findings_counts`). Adding a new dash-able cell breaks pre-existing single-dash assertions. Fix: in the shared test fixture builder give the new field a NON-dash default (e.g. `findings_counts: {critical:2,warning:2,suggestion:1}`), and in each dash-specific test set the OTHER cells non-null so exactly one `‚Äî` remains. Prefer `getByLabelText(...)` for the new cell (e.g. the `FindingsCounts` group's aria-label) ‚Äî it's unique regardless of how many dashes render.
- React (Next 15) logs a runtime console error ‚Äî "Updating a style property during rerender (`borderColor`) when a conflicting property is set (`borderLeftColor`) can lead to styling bugs" ‚Äî whenever a base `style` object sets the `border` SHORTHAND and a spread inline `style` overrides a longhand like `borderColor` on the SAME element (React expands `border` into `borderLeftColor` etc., so the two collide on rerender). Hit when a `FindingsPanel/SeverityFilter` chip merged `{...s.chip, borderColor: active ? meta.c : ...}`. Fix: in the base style object, never use the `border` shorthand if any per-render override touches the border ‚Äî split it into `borderWidth`/`borderStyle`/`borderColor` longhands so only longhands are ever set. Same rule applies to `background`/`margin`/`padding` shorthands paired with longhand overrides.
- FOLLOW-UP / GOTCHA on the above: `borderColor` is ITSELF a shorthand for the four per-side colors, so removing the literal `border` shorthand is NOT enough ‚Äî an element that sets BOTH `borderColor` (all sides) AND `borderLeftColor` (a per-side accent) STILL trips the identical "(`borderColor`) ‚Ä¶ conflicting property ‚Ä¶ (`borderLeftColor`)" warning the moment `borderColor` changes between renders. Real case: `FindingCard/styles.ts` `s.card()` draws a severity-colored left accent (`borderLeftColor: sevColor`, `borderLeftWidth: 3`) plus a focus ring (`borderColor: focused ? sevColor : 'var(--border)'`); toggling `focused` (e.g. j/k nav, or `FindingsPanel` resetting `focusIdx` on a severity-filter click) updated `borderColor` and fired the error in the console. Latent bug, just easy to trigger via the filter. Fix: drop `borderColor` entirely and set the three NON-left sides explicitly ‚Äî `borderTopColor`/`borderRightColor`/`borderBottomColor` (= the focus color) alongside the separate `borderLeftColor`. Rule of thumb: if ANY side color is set individually, set ALL four individually; never mix `borderColor` with a `border<Side>Color`.
- `"GitCompare"` is **not** in the `@devdigest/ui` icon registry ‚Äî referencing it silently resolves to `undefined` and renders nothing. For Smart Diff / risk-ordered-view concepts, use `"Layers"`. Before naming an icon in `PrDetailHeader` or any other component that reads from the UI kit's icon map, verify it exists in `client/src/vendor/ui/` (grep for the name). `"Code"` is already taken by the Files-changed tab.
- The diff-viewer barrel (`client/src/components/diff-viewer/index.ts`) originally exported only `DiffViewer` ‚Äî `parsePatch` and `Line` were NOT exported. Any component that needs `parsePatch` (e.g. `SmartDiffViewer`) must either import from the deep path (`@/components/diff-viewer/helpers`) or add an additive re-export to the barrel first. The additive re-export is preferred (avoids deep internal imports) and was done for Smart Diff: `export { parsePatch, Line } from "./helpers"` added to `index.ts`.
- `vi.useFakeTimers()` + RTL `waitFor` is a deadlock: `waitFor` polls via `setTimeout`/`setInterval` ‚Äî both faked ‚Äî so it never advances and the test times out at 5 000ms. Pattern that works: (a) keep timers REAL during `waitFor` calls; (b) after the action that queues a `setTimeout(0)`, flush it with `await new Promise<void>(r => setTimeout(r, 10))` (a 10ms real wait); (c) then assert. ## Session Notes

- `BlastGraph` previously used `hasContent = callers.length > 0 || rightNodes.length > 0` as its empty gate. After the step-7 fix this became `symbols.length === 0`. As a consequence the BlastGraph tests that previously asserted `SYMBOLS_NO_CALLERS` shows the empty message and no SVG NOW INVERT: `SYMBOLS_NO_CALLERS` renders an SVG. Any future test addition for BlastGraph should follow the new gate: empty message only when `symbols.length === 0`, not when callers are absent.

- BlastRadius endpoint/cron badges must live in `SymbolRow`'s header div (always rendered), NOT inside `CallerItem`. When badges were in `CallerItem` they duplicated once per caller AND disappeared entirely when `callers.length === 0` (the `<ul>` was conditional on caller count). The fix: render `group.endpoints`/`group.crons` badges in the symbol header div regardless of caller count; `CallerItem` becomes link-only. RTL test pattern for "appears exactly once": `screen.getAllByText("POST /auth/login")` + `expect(...).toHaveLength(1)` ‚Äî `getByText` would also catch duplication but throws if missing; `getAllByText` works for both cases.

- RTL `getByText` (and `getAllByText`) uses `getNodeText(node)` which reads **only the node's DIRECT text nodes** (children where `childNode.nodeType === TEXT_NODE`) ‚Äî NOT the concatenated text of all descendants. This means a `<div>` that holds only child `<span>` elements has `getNodeText` = "" even if those spans each contain text. Practical upshot: splitting a stat row into four `<span>` elements (icon + template-literal text per stat) does NOT create multiple-match collisions when you query `getByText(/1 symbols/)` ‚Äî the parent flex container returns "" and the sibling span with "2 callers" doesn't match "/1 symbols/". But a stat item written as `<span><strong>{n}</strong>{" "}{label}</span>` returns only `" label"` (the `<strong>` is an element, not a text node), so the query fails. Fix: render number and label as a single direct text node via template literal ‚Äî `{`${n} ${label}`}` ‚Äî inside the stat item span alongside the icon element. This is the same "collapse to one text node" pattern as the RTL sibling-text rule, applied to WITHIN a stat item rather than across siblings.
