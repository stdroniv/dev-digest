## What Doesn't Work

- Linking a finding's `file:line` to GitHub via `githubBlobUrl` (`/{repo}/blob/{sha}/{file}#L{line}`) 404s in two real cases: (a) FORK PRs â€” `pr.head.sha` lives on the contributor's fork, not the base repo, so `base-repo/blob/{forkSha}` doesn't exist; (b) the demo SEED â€” `seed.ts` uses a fabricated repo (`acme/payments-api`) + fake `head_sha` (`a1b2c3d4e5f6`), so EVERY github.com link 404s regardless of format (a user reported "always 404" here; the format was fine, the data isn't real). Fix: link into the PR's Files-changed diff with `githubPrFileUrl` (`/{repo}/pull/{number}/files#diff-<sha256(path)>R{line}` in `lib/github-urls.ts`) â€” the Files view is owned by the BASE repo, so it resolves for fork PRs. The `#diff-` hash is the SHA-256 hex of the repo-relative path, computed async via Web Crypto in the `usePathShas` hook (`lib/hooks/use-path-shas.ts`); render the bare `/pull/{n}/files` URL immediately and upgrade once the hash resolves (and when `crypto.subtle` is missing â€” jsdom/insecure context â€” keep the bare URL). `R{line}` targets the added/right side; it only anchors precisely if the line is in the diff, but the file still loads otherwise. To verify a real (non-404) link you MUST import a real public repo â€” the seed can't resolve.
- A `position: fixed` portal popover positioned from a ONE-TIME `getBoundingClientRect()` (e.g. `FindingsHoverCard`) DETACHES on scroll: `fixed` pins it to the viewport, so when the page/timeline scrolls the trigger moves but the card stays floating over unrelated content. Must dismiss (or reposition) on `scroll`+`resize` while open. Two subtleties: (a) register the `scroll` listener in the CAPTURE phase (`window.addEventListener('scroll', fn, true)`) â€” `scroll` does NOT bubble, and the content may live in a NESTED `overflow:auto` scroller a non-capture window listener would miss; (b) IGNORE scrolls originating inside the popover's own `overflow:auto` list via `popoverRef.current?.contains(e.target)`, or scrolling a long findings list self-dismisses the card. Regression test: open via `mouseEnter`, `fireEvent.scroll(window)`, assert it's gone (`RunHistory.test.tsx`).

## Codebase Patterns

- The PR-list FINDINGS popover re-derives its finding list CLIENT-SIDE in `components/FindingsCounts/aggregate.ts` (`aggregateLatestPerAgent`), which is a SECOND copy of the rule the server uses to compute the `findings_counts` badge in `server/src/modules/pulls/routes.ts` (latest review per `(pr, agentId)`, `kind === 'review'` only, dismissed findings INCLUDED, sorted severityâ†’confidence). The badge total and the popover list length only match because both apply the identical rule â€” change one and you MUST change the other, or the header "N findings" disagrees with the rows shown. The popover deliberately reuses the existing `GET /pulls/:id/reviews` endpoint (no new endpoint / contract field) and fetches lazily on hover, which is why the client must re-aggregate rather than read a server-prepared list.
- A hover popover/dropdown anchored inside the PR-list table will be CLIPPED: `styles.ts` `tableCard` sets `overflow: hidden` (needed for its rounded corners), so an absolutely-positioned child is cut at the card edge. Render the floating layer via `createPortal(node, document.body)` with `position: fixed` coords from the trigger's `getBoundingClientRect()` instead (see `PRRow/FindingsCell.tsx`). Keep it open across the triggerâ†’popover gap with a short close-delay timer that both elements' `onMouseEnter` cancel, and `stopPropagation` on the popover so clicks don't fire the row's navigate-onClick.
- A PR-list row component (`PRRow`) now fetches lazily on hover via `usePrReviews` (TanStack Query). Any test that renders `PRRow` must wrap it in a `QueryClientProvider` or `useQuery` throws "No QueryClient set" â€” even though the query is `enabled:false` until hovered (so no fetch mock is needed for the closed/default state). Mirror `agents/_components/AgentCard/AgentCard.test.tsx`.
- The PR-list table is a POSITIONAL CSS grid, not a column-config map. Adding/removing a column means editing THREE places in sync in `src/app/repos/[repoId]/pulls/`: `constants.ts` `GRID` (add a track width â€” the string order = visual order), `constants.ts` `COLUMN_KEYS` (the header auto-renders from this array in `page.tsx` via `t("list.columns.<key>")`), and `_components/PRRow/PRRow.tsx` (insert the cell `<div>` at the MATCHING index â€” cells are emitted in source order with no key binding). Miss one and headers silently misalign from cells. Reuse `SEV`/`Icon` from `@devdigest/ui` (re-exported via the primitives barrel) for severity color+icon instead of redefining maps.
- "Findings by severity" is TWO separate features that look identical but live at different altitudes â€” don't conflate them (a reviewer flagged exactly this). (a) PR-LIST badge: `components/FindingsCounts/*` + `PRRow/FindingsCell.tsx` â€” read-only counters + hover card over the cross-agent aggregate (`aggregateLatestPerAgent`). (b) PR-DETAIL inline filter: `pulls/[number]/_components/FindingsPanel/` â€” clickable severity chips in the panel toolbar that filter that ONE review run's list in place. The detail filter is the asked-for feature; the list badge is a bonus. Both reuse the same `SEV` (color/bg/icon/label) + `Icon` tokens from `@devdigest/ui` â€” never redefine severity maps. Detail-filter wiring: `sevFilter` state in `FindingsPanel`; `visibleFindings(findings, hideLow, sevFilter)` (in `helpers.ts`) applies hide-low THEN the severity filter THEN the severity sort; `countBySeverity()` (pure) tallies chips over the POST-hide-low set so a chip's number always equals the rows shown when it's selected; reset `focusIdx` to 0 on filter change so j/k nav never points past the shrunken list; zero-count chips render `disabled`. Chip click toggles (click active chip â†’ clear). Tested with `fireEvent.click` (NO `@testing-library/user-event` in this package) querying chips by their i18n `aria-label` `panel.filterSeverity` ("Show only {severity} ({count})").

## Codebase Patterns

- Deep-linking from the PR-list findings tooltip to a specific finding's card on the detail page: build `/repos/{repoId}/pulls/{number}?tab=findings#finding-{id}` (the card renders `data-finding-id={f.id}`). Read the finding id from `window.location.hash` in a mount `useEffect` â€” `useSearchParams()` does NOT include the hash, so a `?finding=` query would be simpler but the existing `?tab=` plumbing already drives the tab. Threading: `page.tsx` â†’ `FindingsTab` â†’ `ReviewRunAccordion` â†’ `FindingsPanel`. CRUCIAL: a finding lives inside a COLLAPSIBLE `ReviewRunAccordion`, and `FindingsPanel` (hence the `FindingCard`) only mounts when the accordion is OPEN â€” so `ReviewRunAccordion` must detect `review.findings.some(f => f.id === focusFindingId)` and `setOpen(true)` BEFORE `FindingsPanel`'s scroll-into-view effect can find the element. Inside `FindingsPanel`, the focus effect keys on `[focusFindingId, shown]` (not `[]`) so it re-runs once the list renders. When a row links in-app, keep the `file:line` GitHub `<a>` a SIBLING of the title `<Link>`, never nested â€” nested `<a>`/`<Link>` is invalid HTML (the title row is the in-app target; the file ref opens GitHub in a new tab).
- The detail-page finding GitHub link needs the PR NUMBER, not the head sha (the old `githubBlobUrl` used `headSha`). When switching `FindingCard`/`FindingsPanel`/`FindingsTab`/`ReviewRunAccordion` to `githubPrFileUrl`, replace the whole `headSha` prop chain with `prNumber` (sourced from `pr.number` in `page.tsx`) rather than passing both â€” `headSha` was link-only plumbing.

- Testing a hover popover that `createPortal`s to `document.body` (e.g. `FindingsHoverCard`) in jsdom: (1) `mouseEnter` does NOT bubble, so `fireEvent.mouseEnter` must target the exact element with the handler â€” the wrapper div, reachable as `screen.getByLabelText("â€¦counts aria-labelâ€¦").parentElement` â€” NOT the inner counters node. No need to stub `getBoundingClientRect`. See `RunHistory.test.tsx` "findings hover popover".

- There is **no ESLint or Prettier anywhere** â€” not in `client/` and not at the repo root (no `eslint.config.*`/`.eslintrc`/`.prettierrc`, no eslint/prettier deps). ## Recurring Errors & Fixes

- A cell component rendered by `PRRow` (e.g. `FindingsCell`) that calls `useParams()` crashes `PRRow.test.tsx` â€” that test does `vi.mock("next/navigation", () => ({ useRouter: ... }))` and provides ONLY `useRouter`, so `useParams` is `undefined` â†’ throws inside the render. Don't reach for `useParams` in PR-row cells: `PRRow` already RECEIVES `repoId` as a prop, so thread it down (`<FindingsCell pr={pr} repoId={repoId} />`). (`useActiveRepo()` is safe in that test â€” its context has a default value â€” so reading `activeRepo.full_name` from it does not need a provider.)
- RTL `getByText(/whole string/)` fails ("Unable to find an element") when JSX interpolates siblings: `{a} tok Â· {formatUsd(b)}` renders THREE text nodes ("9,731", " tok Â· ", "$0.012"), and `getByText` matches per-node by default. Fix in the component (not the test) by collapsing to one text node with a template literal: `{`${a} tok Â· ${formatUsd(b)}`}`. Done for the run cost line in `RunHistory.tsx`.
- `getByText("â€”")` throws "Found multiple elements" once a row can render the em-dash placeholder in MORE than one cell (e.g. `PRRow` shows `â€”` for both null `cost_usd` and null `findings_counts`). Adding a new dash-able cell breaks pre-existing single-dash assertions. Fix: in the shared test fixture builder give the new field a NON-dash default (e.g. `findings_counts: {critical:2,warning:2,suggestion:1}`), and in each dash-specific test set the OTHER cells non-null so exactly one `â€”` remains. Prefer `getByLabelText(...)` for the new cell (e.g. the `FindingsCounts` group's aria-label) â€” it's unique regardless of how many dashes render.
- React (Next 15) logs a runtime console error â€” "Updating a style property during rerender (`borderColor`) when a conflicting property is set (`borderLeftColor`) can lead to styling bugs" â€” whenever a base `style` object sets the `border` SHORTHAND and a spread inline `style` overrides a longhand like `borderColor` on the SAME element (React expands `border` into `borderLeftColor` etc., so the two collide on rerender). Hit when a `FindingsPanel/SeverityFilter` chip merged `{...s.chip, borderColor: active ? meta.c : ...}`. Fix: in the base style object, never use the `border` shorthand if any per-render override touches the border â€” split it into `borderWidth`/`borderStyle`/`borderColor` longhands so only longhands are ever set. Same rule applies to `background`/`margin`/`padding` shorthands paired with longhand overrides.
- FOLLOW-UP / GOTCHA on the above: `borderColor` is ITSELF a shorthand for the four per-side colors, so removing the literal `border` shorthand is NOT enough â€” an element that sets BOTH `borderColor` (all sides) AND `borderLeftColor` (a per-side accent) STILL trips the identical "(`borderColor`) â€¦ conflicting property â€¦ (`borderLeftColor`)" warning the moment `borderColor` changes between renders. Real case: `FindingCard/styles.ts` `s.card()` draws a severity-colored left accent (`borderLeftColor: sevColor`, `borderLeftWidth: 3`) plus a focus ring (`borderColor: focused ? sevColor : 'var(--border)'`); toggling `focused` (e.g. j/k nav, or `FindingsPanel` resetting `focusIdx` on a severity-filter click) updated `borderColor` and fired the error in the console. Latent bug, just easy to trigger via the filter. Fix: drop `borderColor` entirely and set the three NON-left sides explicitly â€” `borderTopColor`/`borderRightColor`/`borderBottomColor` (= the focus color) alongside the separate `borderLeftColor`. Rule of thumb: if ANY side color is set individually, set ALL four individually; never mix `borderColor` with a `border<Side>Color`.
