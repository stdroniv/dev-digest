# reviewer-core — Engineering Insights

Append-only log of non-obvious, hard-won lessons for this module
(`@devdigest/reviewer-core`). Managed by the `engineering-insights` skill. Add each
entry under one section; keep it actionable cold; never edit or delete existing entries.

## What Works

- The false-negative re-sample guard in `reviewPullRequest` (`src/review/run.ts`) MUST perturb each extra draw — bump `temperature` (default 0.4) AND offset the seed (`seed + i + 1`). Re-sending the same messages at `temperature: 0` + the same seed just reproduces the identical lazy-empty result (especially with the provider pinning from the determinism fix), silently defeating the guard. The guard is also scoped to `mode === 'single-pass'` and to `merged.findings.length === 0`, and the dedup (`dedupeFindings` in `reduce.ts`) is applied ONLY on the re-sample branch — so the normal single-sample path stays byte-identical to before (the repo's "omitted slot → identical behavior" contract). Off by default via `resampleOnEmpty ?? 0`.

## What Doesn't Work

- The grounding gate (`src/grounding.ts`) is an ASYMMETRIC safety net: it only DROPS findings whose lines don't intersect a diff hunk (defends against false positives / hallucinated locations). It has ZERO defense against false NEGATIVES — when the model lazily returns `findings: []`, nothing recovers them, and `scoreFromFindings([])` (`src/review/reduce.ts`) mechanically returns 100 → a clean "approve". A single low-effort sample silently passes a buggy PR. Don't treat an empty findings list as trustworthy; if you need a real merge gate, add an explicit guard (re-sample on empty / self-consistency over N samples / worst-verdict union) — grounding will NOT catch this for you.
- A trusted instruction that CONSUMES attacker-influenced "intent/scope" must never be allowed to gate review depth. The Intent Layer injects a classifier-derived `## PR Intent` block (summary + in/out-of-scope) into the review prompt with a trusted `SCOPE_DISCIPLINE_RULE` (`src/prompt.ts`). The intent is derived from the PR title/body/linked-issue — all author-controlled — so a rule like the originally-specced "review only within the stated intent; one signal finding, not twenty" is a SUPPRESSION VECTOR: a crafted PR can mark a vulnerable area `out_of_scope` to thin or silence findings there. The fix (and the invariant for any future scope/intent slot): the trusted rule must mirror `INJECTION_GUARD` — stated scope may inform a finding's *rationale/ordering* only and can NEVER reduce the count or severity of a real defect ("report it regardless of stated scope"). Relatedly, the classifier's OWN call (`src/intent/classify.ts` `CLASSIFY_SYSTEM`) must carry `INJECTION_GUARD` too: `wrapUntrusted()`-fencing the PR content is *decorative* without the matching trusted rule telling the model the fenced text is data, not instructions. `INJECTION_GUARD` is exported from `prompt.ts` — import it, don't re-author the wording.
- `temperature: 0` (`src/llm/openrouter.ts:72`) does NOT make reviews reproducible. A byte-identical prompt (verified via SHA over every `prompt_assembly` section) produced 2 findings / 1643 output tokens one run and 0 findings / 131 tokens the next, same agent + same PR + same model. Causes: OpenRouter routes a model id across multiple upstream hosts/quantizations with no provider pinning, no `seed` is sent, and MoE models (e.g. `deepseek/deepseek-v4-flash`) route experts non-deterministically under server-side batching. `…-flash`/distilled tiers are especially prone to short lazy completions. To reduce drift: send a `seed` and pin `provider: { allow_fallbacks: false }` in the `chat.completions.create` call, and don't gate merges on a flash-tier model.
- `OpenRouterProvider` (`src/llm/openrouter.ts`) historically implemented ONLY `completeStructured` and stubbed `complete()` + `embed()` to `throw new Error('OpenRouterProvider only implements completeStructured')` (the `NOT_SUPPORTED` const), on the documented assumption "only completeStructured is needed by reviewPullRequest." That broke the blast-radius summary: `server/src/modules/blast/summary.service.ts` calls `llm.complete()` (free-text, one paragraph), and `resolveCheapLlm()` there falls through `openai → anthropic → openrouter`, so an OpenRouter-only config (no OpenAI/Anthropic key) hit the stub and crashed the "Explain impact" button with that exact error string. Fix: implement `complete()` for real — OpenRouter is OpenAI-compatible, so it's a plain `chat.completions.create` with NO `response_format`/parse-with-repair loop, reusing the same no-choices guard + OpenRouter `usage.cost` accounting (falling back to the injected `estimateCost`) as `completeStructured`. LESSON: all three providers (`openai`/`anthropic`/`openrouter`) must satisfy the FULL `LLMProvider` interface — before adding any new free-text (`complete`) or embedding (`embed`) feature, verify every provider actually implements that method, don't assume `completeStructured`-only. `embed()` is still a deliberate stub on OpenRouterProvider (nothing routes embeddings there yet) — wire it if a feature ever does.

## Codebase Patterns

- The `skills` prompt slot (`assemblePrompt` in `src/prompt.ts`) is TRUSTED content: skill bodies are joined with `\n\n` under `## Skills / rules` and are deliberately NOT `wrapUntrusted()`-fenced (unlike the diff, PR body, repo-map, specs). The trust boundary is enforced upstream — the server's run-executor injects only `enabled` skills, and imported/community skills are created `enabled:false` until a human vets them. So if you add a new skill-like slot, decide trust explicitly: vetted house rules → plain; anything author/third-party-controlled → `wrapUntrusted`.
- All THREE of a review's headline signals — score, persisted verdict, and the GitHub review event — are DERIVED from the grounded findings post-gate, NOT taken from the model's self-reported `Review` fields (which drift). The model's `Review.verdict` field exists and looks authoritative but is OVERRIDDEN: `reviewPullRequest` (`src/review/run.ts`, the grounding-return) sets `score: scoreFromFindings(ground.kept)` AND `verdict: verdictFromFindings(ground.kept)` (`src/output/to-review.ts`), and `toReviewPayload` maps the SAME `verdictFromFindings` to its CI event. `verdictFromFindings(findings, failOn='critical')` = `0 findings → approve; gateTriggered → request_changes; else comment` — it reuses `gateTriggered`, so don't re-author the severity logic. Regression history: the verdict was the ONE field left un-derived (it rode along via `...merged`), so a model could nitpick in prose, self-report `request_changes` + score 50, file zero (or only hallucinated, later-dropped) findings, and the studio's `ReviewRunAccordion` rendered the raw verdict → the contradictory "100 / 0 findings / request changes" card. If you add any new findings-derived signal, derive it here at the grounding return alongside score/verdict — never trust `merged.*` (pre-grounding, model-reported).
- Convention citation grounding (`src/conventions/verify.ts` `verifyConventions`/`locateSnippet`) is a FUZZY gate, not a literal-string check: it matches the snippet's first NON-BLANK line via whitespace-normalized substring (`normalize` collapses `\s+`, then `.includes()`) against the candidate's evidence file, then RE-DERIVES the line range from where that first line lands (overwriting the model's reported start/end lines). Two consequences when debugging dropped/surviving candidates: (1) a paraphrased-but-close first line can still pass — it's not exact equality; (2) it validates against whatever `files` Map the caller passes, and the server (`modules/conventions/service.ts`) passes the SAME in-memory `samples` already sliced to `MAX_SAMPLE_CHARS` (8000), NOT a fresh disk read — so a real citation whose first line sits beyond ~char 8000 of a large file can't ground and is silently dropped. Separately: there is NO confidence-threshold filter anywhere — `ConventionDraft.confidence` is `z.number().min(0).max(1)` with no floor and nothing drops low-confidence drafts, so a 0.0-confidence candidate persists unless the extraction prompt itself suppresses it.

- **`reviewer-core/tsconfig.json` only sets `"include": ["src/**/*.ts"]` — the top-level `test/` directory (where most existing `*.test.ts` files live) is NOT typechecked by `pnpm typecheck` (`tsc --noEmit -p tsconfig.json`) at all.** Those files are only ever transpiled (not type-checked) by `pnpm test` (vitest/esbuild). Consequence: a test file placed under `test/` can carry type errors indefinitely without `pnpm typecheck` ever catching them. When a plan/task explicitly asks for a NEW test file under `src/<module>/*.test.ts` (colocated, not `test/`) rather than the existing convention, that's deliberate — it's the only way to get that file real typecheck coverage, and it still runs fine under vitest (`vitest.config.ts`'s `include` already covers both `test/**/*.test.ts` and `src/**/*.test.ts`). It can freely import server-only test helpers (e.g. `MockLLMProvider` from `../../../server/src/adapters/mocks.js`) via a relative path — that's the established cross-package pattern here (see the pre-existing `test/risks.test.ts`), and reviewer-core's `tsconfig.json` `paths` + `moduleResolution: "Bundler"` resolve it fine even though the target file lives physically outside `include`.

## Tool & Library Notes

- To debug "agent verdict regressed but nothing changed", compare the two runs' persisted prompts directly — don't trust the human-readable summaries (they describe the OUTPUT, which can diverge wildly from identical input). Traces live in Postgres, not the filesystem: `select trace from run_traces where run_id='<uuid>'` (jsonb). SHA-256 each `prompt_assembly` section (`user`, `system`, `skills`, `callers`, `repo_map`, `pr_description`) across both runs; if `tokens_in` matches and all hashes match, the input is byte-identical and the regression is pure model non-determinism, not your prompt/skills/diff. The `runs/<id>/trace` path users quote is a client route, not a file.

## Recurring Errors & Fixes

- A "no diff bodies / hunk-headers-only" defensive filter must drop ALL THREE unified-diff body-line prefixes — a leading **space** (context), `+` (added), and `-` (removed) — not just `+`/`-`. Context lines start with a space and carry REAL unchanged source code, so a filter like `/^[+-]/` silently leaks them to the LLM. Use `/^[ +-]/` (see `stripCodeLines` in `src/intent/classify.ts`), which keeps exactly the legitimate shapes the server's `buildHunkHeadersBlock` emits (bare path lines, `@@ … @@` headers, blank separators — none of which start with space/`+`/`-`). Verify with a test whose input includes a space-prefixed context line and asserts it's absent from the assembled prompt.
- `wrapUntrusted(label, content)` (`src/prompt.ts`) only escaped `content`'s literal `</untrusted>` sequences — NOT `label`. Every pre-existing caller passes a fixed, code-controlled label (`'diff'`, `'pr-description'`, `'pr-intent'`, `'repo-map'`, `'callers'`), so this was safe by accident. The `specs` slot (Project Context) broke that invariant: it labels each block by `doc.path`, an attacker-influenceable STORED value (a persisted document path). A path containing a literal `</untrusted>` could close the wrapper early and inject a fake one before the real content, defeating the model's textual fencing. Fix: escape `label` with the SAME `.replaceAll('</untrusted>', '<\\/untrusted>')` treatment as `content`, before interpolating into `source="${label}"`. Lesson: any time you add a NEW `wrapUntrusted` call site whose label is anything other than a hardcoded string literal, treat the label as untrusted too — it's not automatically covered by the content-escaping.
