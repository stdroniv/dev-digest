# server ŌĆö Engineering Insights

Append-only log of non-obvious, hard-won lessons for this module (`@devdigest/api`,
includes `repo-intel`). Managed by the `engineering-insights` skill. Add each entry
under one section; keep it actionable cold; never edit or delete existing entries.

## What Works

- **Anchoring `cloneDir` to `import.meta.url` fixes the MCP vs API blast divergence with zero runtime cost.** Replace `resolve(process.cwd(), cloneDirRaw)` with `resolve(dirname(fileURLToPath(import.meta.url)), '..', '..')` (= `SERVER_ROOT`) in `platform/config.ts`. Because `import.meta.url` always resolves to the physical file on disk regardless of cwd, every process that imports `config.ts` (API at `cwd=server/`, MCP at `cwd=mcp/`, any future consumer) derives the SAME `cloneDir`. Two details: (1) the default changes from the undocumented `~/.devdigest/workspace` to `<server>/clones`, aligning with `.env.example` and README; (2) an absolute `DEVDIGEST_CLONE_DIR` is still returned verbatim ŌĆö the anchor only affects relative values. Regression guard in `test/config.test.ts`: call `process.chdir(os.tmpdir())` inside the test and assert `loadConfig({}).cloneDir` is the same before and after the chdir.

- **T8 (`db/seed-ci.ts`) reuses `modules/ci/helpers.ts`'s pure `computeConfigHash`/`resolveAgentSlug`/`toBundleAgent`/`blockersFromCounts` directly from the `db/` layer to make a seeded "current" `ci_installations` row genuinely non-drifted.** `CiService.listInstallations`'s `update_available` is `installed_config_hash !== computeConfigHash(liveAgent, liveLinkedSkillNames, slug)` recomputed fresh at READ time ŌĆö a seed can't fake a matching hash with a hardcoded string, it has to compute the SAME sha256 over the SAME live agent row + live linked-skill names it just seeded (with `otherExportedAgentNames` safely assumed empty, since the seed is the only thing granting this agent a CI installation in a fresh workspace). This makes the "not drifted" installation self-repairing across `pnpm db:seed` reruns even if the demo agent's system prompt changes later ŌĆö no hardcoded hash to keep in sync. The DRIFT-eligible installation needs no such care: any string that isn't a real sha256 hex digest (e.g. `'seed-stale-config-hash'`) guarantees `update_available: true` forever. `db/*.ts` importing a pure, I/O-free helper from `modules/<x>/helpers.ts` is a reasonable exception to the usual dbŌåÆmodules layering direction ŌĆö same precedent as `seed.ts` already importing `platform/reviewer-prompts.ts`.
- **Idempotent "insert a row, and ALSO insert a second row in a different table but only if the first insert was NEW" (T8's `ci_runs` ŌåÆ matching `agent_runs(source='ci')`) is cheaply solved by reading `.onConflictDoNothing({ target: [...] }).returning()`'s result, not by a separate existence check.** `agent_runs` has no FK/unique key back to a specific `ci_runs` row (confirmed by server INSIGHTS.md:98's reconcile note), so a seed re-run can't ask "does the matching `agent_runs` row already exist?" directly. Instead: the `ci_runs` insert is keyed by the SAME `(ci_installation_id, actions_run_id)` unique index reconcile relies on; `onConflictDoNothing(...).returning({id: ...})` returns the row ONLY when it was actually inserted (empty array on a conflict) ŌĆö use that emptiness as the "already seeded, skip the `agent_runs` insert too" signal. Exactly mirrors `CiRepository.upsertCiRun`'s `justBecameTerminal` gate (server INSIGHTS.md:98) but needs no extra `SELECT` beforehand.

- **`SimpleGitClient.ensureHistory()` lazy deepen for prior-PR history: the in-process `Set<string>` cache is LOAD-BEARING, not optional.** `git fetch --shallow-since="2 years ago"` deepens a shallow clone but the clone STILL reports `git rev-parse --is-shallow-repository = true` after the fetch (the shallow marker stays in `.git/shallow` until `--unshallow`). Without the `private deepened = new Set<string>()` cache keyed on the absolute clone path, every `log()` call ŌĆö including the 25 per-file calls inside one `HistoryService.getPriorPrs` request ŌĆö would re-run the `rev-parse` + `fetch` on every invocation. The cache also means offline/auth failures (fetch throws ŌåÆ not added to `deepened`) are automatically retried on the next request, with no extra try/catch needed. Detect shallow clones with `rev-parse --is-shallow-repository` (git Ōēź 2.15); for older git, fall back to checking `existsSync('.git/shallow')`.

## What Doesn't Work

- The denormalized `agent_runs.findings_count` / `blockers` columns DRIFT from reality ŌĆö they can be `0`/null while the run's review actually has findings ŌĆö so a Timeline run row built from them reads "0 finding(s)" + a green "approved" badge while the Review-runs accordion below (reading `review.findings`) correctly shows the findings. `completeAgentRun` does set `findingsCount` at run end, but it's unreliable cold (stale rows, seed data, any path that adds/changes findings after completion). Don't trust those columns on read: in `run.repo.ts` `listRunsForPull` the per-severity `findings_counts` was ALREADY computed fresh from `findings ŌåÆ reviews.run_id`, but `findings_count`/`blockers` were still read from the denorm columns ŌĆö derive those two from the same fresh join instead (the client `RunHistory.outcomeOf` keys the badge off `findings_count`+`blockers`, so this also fixes the wrong "approved" color with no client change). Regression test pattern: insert a run with `findingsCount: 0, blockers: null` but a review with a CRITICAL + WARNING finding, then assert `GET /pulls/:id/runs` returns `findings_count: 2, blockers: 1` (`findings-counts.it.test.ts`).
- The demo seed (`server/src/db/seed.ts`) creates PR #482's sample review DECOUPLED from the timeline run: it inserts the `review` (+ findings) with NO `run_id` and NO `agent_id`, then later inserts the `agent_runs` row separately (only `findings_count`/`blockers` are denormalized onto it). So any feature that attributes a run's data via `reviews.run_id` (e.g. per-run `findings_counts` in `run.repo.ts` `listRunsForPull`, which joins `findings ŌåÆ reviews ON reviews.run_id = agent_run.id`) gets NOTHING for seeded runs ŌåÆ null ŌåÆ the UI shows the em-dash fallback. Real runs are fine (`run-executor` sets `review.runId`); only the seed is unrealistic. Fix: after creating the seeded run, `db.update(t.reviews).set({ runId, agentId })` for that PR's review ŌĆö place it OUTSIDE the `!existingRun` guard so re-running `pnpm db:seed` repairs an already-seeded DB (idempotent). Linking `agent_id` also fixes the Review-runs row showing a generic "Agent" instead of the agent name.
- Adding a new contract field as `.nullable()` (REQUIRED, value-or-null) instead of `.nullish()` (optional) silently breaks existing tests that build the object literally. Concretely: making `RunSummary.findings_counts` `.nullable()` failed `contracts.test.ts`'s `RunSummary.parse({...})` fixture and the client `run()`/`pr()` test builders with a Zod "Required" error, because those omit the new key. Fix: either use `.nullish()`, or update every literal fixture/builder to include the field. (`PrMeta` used `.nullish()` and needed no fixture change.)
- `GitClient.defaultBranch()` (`adapters/git/simple-git.ts`) has a three-step detection chain: (1) `git rev-parse --abbrev-ref origin/HEAD` ŌåÆ fails silently when `origin/HEAD` is not set on the remote/shallow clone; (2) `git rev-parse --abbrev-ref HEAD` ŌåÆ returns the literal string `'HEAD'` (not a branch name) whenever the clone is in detached HEAD state; (3) hardcoded fallback `'main'` ŌåÆ this fires for `master` repos in the detached-HEAD + no-origin/HEAD combination, silently writing `'main'` to the DB. Fix added to `defaultBranch()`: before returning `'main'`, try `git ls-remote --symref origin HEAD` and parse with `/^ref:\s+refs\/heads\/(\S+)\s+HEAD/m` ŌĆö this queries the remote authoritatively and is acceptable inside a clone-job context. Any repo already in the DB with the wrong `'main'` default needs a Refresh to re-run `runCloneJob` and re-detect.
- `repos.default_branch` defaulted to `'main'` in the schema (`db/schema/repos.ts`) and was NEVER fetched from the real repo ŌĆö `RepoRepository.insert` didn't set it. This silently broke TWO things for any repo whose real default is `master` (e.g. `actualbudget/actual`, `Staffjoy/v2`, `tw93/Maple`): (1) the client GitHub blob deep-links (`ConventionsWorkspace` `repoRef = activeRepo.default_branch` ŌåÆ `githubBlobUrl`) 404'd, and (2) repo-intel "re-analyze" (`repo-intel/service.ts` `sync(ref, repo.defaultBranch)` ŌåÆ `git reset --hard origin/main`) silently failed. Fix: `runCloneJob` (`modules/repos/service.ts`) now calls the new `GitClient.defaultBranch(repo)` (impl: `git rev-parse --abbrev-ref HEAD`, since `git clone` without `--branch` checks out the remote default) and persists it via `updateClonePath(repoId, path, defaultBranch)`. KEY GOTCHA: existing repo rows keep `'main'` ŌĆö they only backfill on a **Refresh / re-clone** (the conventions page "Re-scan" runs `extract`, NOT a clone, so it does NOT fix the branch). When debugging a stale `default_branch`, re-clone the repo, don't re-scan.
- The `GET/PUT /settings` routes need NO route-schema change to add a brand-new settings
  key (e.g. `root_folders`, T4). `Settings`/`SettingsUpdate` (`vendor/shared/contracts/
  platform.ts`) are built as `SettingsKnown.passthrough()` (not `.strict()`), so a Zod
  object's `.partial()` on a passthrough schema stays passthrough ŌĆö arbitrary extra keys in
  the `PUT /settings` body sail through validation untouched, get looped over in
  `Object.entries(body)` and upserted, and `GET /settings` returns the *raw* folded rows
  (`rowsToSettings`, no `.parse` at all) ŌĆö so an unvalidated new key round-trips end-to-end
  with zero route code changes. This means "does the route need a schema change to accept
  key X" is the WRONG question for this endpoint; the real design decision is whether X's
  *shape* should be validated at write time (like `feature_models: z.record(...)` in
  `SettingsKnown`, which does add real structural validation) or left permissive and
  validated only at the READ site via a dedicated `resolveX`/`getX` helper (like
  `getRootFolders` in `modules/settings/root-folders.ts`, which `safeParse`s and falls back
  to a default on garbage). Prefer the read-site-validates pattern for anything read by a
  narrow set of consumers (here: the documents-discovery service) ŌĆö it avoids touching the
  vendored contract (`*/src/vendor/shared` in three places) for a field only one caller needs.
- The Conventions Extractor has NO `replaceAll`/delete step, so re-scanning a repo ACCUMULATES duplicate candidates. `ConventionsService.extract()` (`modules/conventions/service.ts`) generates a fresh `runId`, calls `ConventionsRepository.insertMany()`, and NEVER deletes prior rows; `listByRepo()` returns every non-`rejected` row across ALL runs (it does not filter to the latest `runId`). So each "Re-scan" inserts a whole new batch of `pending` candidates on top of the old ones ŌĆö the UI fills with duplicates and previously-accepted rules get re-proposed. Note there is no `server/src/modules/conventions/extractor.ts` despite the name: orchestration lives in `service.ts`, the pure steps in `reviewer-core/src/conventions/{extract,verify,assemble}.ts`. To make re-scan idempotent, delete the repo's prior non-accepted candidates (or all rows for older `runId`s) inside `extract()` before `insertMany`, or add a real `replaceAll` to the repository.
- Blast Radius reports `0 endpoints ┬Ę 0 cron` for Next.js App-Router PRs (e.g. cal.com `apps/web/app/api/cron/**/route.ts`) because of TWO compounding gaps. (1) `extractEndpoints`/`extractCrons` (`adapters/codeindex/extract.ts`) only match Express/Fastify shapes (`app.get('/x')`, `router.post(...)`, `route({method,url})`) and cron-expression/`register('kind')` calls ŌĆö they NEVER recognize `export async function GET|POST(...)` in a `route.ts`, where the path comes from the FILE PATH (`app/api/cron/foo/route.ts` ŌåÆ `/api/cron/foo`). So `file_facts` is empty for those files. (2) Even with facts present, `tryPersistentBlast` (`repo-intel/service.ts`) reads `getFileFacts(repoId, callerFiles)` ŌĆö CALLER files only. A route handler IS the endpoint and has 0 cross-file callers (the framework invokes it via file-based routing), so `callerFiles` is empty and its OWN facts are never read. To surface route endpoints you must BOTH (a) make `extractEndpoints`/`extractCrons` path-aware (take `relPath`, derive the route from the path, emit one entry per exported verb, flag `/api/cron/**`) ŌĆö then REINDEX, since `file_facts` is built at index time, not read time ŌĆö AND (b) attribute the changed files' own facts, not just their callers'. (This is distinct from the already-documented `getResolvedCallers` precision-over-recall "0 callers is expected" note in `client/INSIGHTS.md`.)
- Blast Radius shows **0 callers for default-exported-by-separate-statement symbols** (e.g. cal.com `getAppCategories` in PR #29558), even though they have real cross-file callers AND `references` rows exist. Root cause is in `parseSymbols` (`adapters/astgrep/index.ts`): the `exported` flag is set by `unwrapExport` (handles INLINE forms ŌĆö `export default function/class`, `export const x`) and a re-export pass (handles `export { x }` clauses ONLY). NEITHER handles `export default <identifier>`: for `const getAppCategories = (...) => {ŌĆ”}` followed by a SEPARATE `export default getAppCategories;`, the `export default` node's payload is a bare `identifier`, `handleDecl` ignores it, and the original `const` keeps `exported: false`. Proven empirically: `parseSymbols` gives `exported:true` for `export default function`/`export const arrow`, but `exported:false` for `const X=ŌĆ”; export default X;`. That one flag defeats BOTH caller paths: edge resolution (`resolveReferences` SQL joins `ŌĆ” AND s.exported = true`, `repository.ts:412`) AND the Tier-1 name-unique fallback (`getUniqueExportFiles` filters `eq(t.symbols.exported, true)`, `repository.ts:619`). Tell-tale in the UI: inline `export default function Foo()` symbols DO resolve callers while `const X=ŌĆ”; export default X` siblings show 0. This is the most common export form in Next.js/React monorepos, and it silently defeats the `blast-radius-caller-resolution-gaps` plan's own Tier-1 fallback (the plan chased `file_edges`/cross-package/type-refs and missed this). Fix: back-patch `exported` for `export default <identifier>` in `parseSymbols` (mirror the `export { X }` re-export pass: an `export_statement` with a `default` child + a bare `identifier` payload ŌåÆ mark the prior same-named symbol exported), add a regression case to `test/astgrep.test.ts`, and bump `INDEXER_VERSION` (`repo-intel/constants.ts`) ŌĆö `symbols.exported` is written at index time so existing repos need a reindex. Residual limitation (separate, deeper): a default import bound to a DIFFERENT local name (`import gac from "ŌĆ”/getAppCategories"; gac()`) records the reference as `gac`, so it still won't match the declared `getAppCategories`.
- Blast Radius / repo-intel reads resolve the clone root in a **cwd-dependent** way that DISAGREES with the absolute path stored in the DB ŌĆö so the SAME `BlastService.getBlast` returns symbols in the API but `no_data`/0 symbols in the MCP server (or any script). `RipgrepCodeIndex.symbols()/references()/grep()` (`adapters/codeindex/ripgrep.ts:46-126`) compute the root via `git.clonePathFor(repo)` = `join(config.cloneDir, owner, name)` (`adapters/git/simple-git.ts:37-39`); they IGNORE `repos.clone_path` (the absolute path written at clone time, also loaded as `repo.clonePath` in `getBlastRadius` and used by `readClone`). `config.cloneDir` (`platform/config.ts:64-80`) resolves a relative `DEVDIGEST_CLONE_DIR` against `process.cwd()` (line 68), and `import 'dotenv/config'` (line 1) loads `.env` from cwd ONLY. Result: API (cwd=`server/`, `server/.env` has `DEVDIGEST_CLONE_DIR=./clones`) ŌåÆ `server/clones/<owner>/<name>` Ō£ō; MCP server (cwd pinned to `mcp/` by `bin/devdigest-mcp.mjs`, no `mcp/.env`, `.mcp.json` passes only `DATABASE_URL`) ŌåÆ falls back to the default `~/.devdigest/workspace` (which doesn't exist) ŌåÆ the directory walk finds nothing ŌåÆ `{changedSymbols:[], degraded:true, reason:'no_data'}`. Proven by A/B with identical bootstrap code: 0 symbols under MCP defaults vs calcom/cal.diy#29557ŌåÆ2, #28926ŌåÆ70 once `DEVDIGEST_CLONE_DIR` pointed at `server/clones` (matching the live `GET /pulls/:id/blast` exactly). Robust fix: read the persisted `repo.clonePath` instead of re-deriving via `clonePathFor`, or anchor relative `cloneDir` to a stable dir (not `process.cwd()`) AND align the default (`./clones` vs `~/.devdigest/workspace` disagree). Affects every cwd-sensitive consumer (MCP, cron, ad-hoc scripts), not just blast.
- `CLONE_DEPTH = 1` (`modules/repos/constants.ts:9`, used in `modules/repos/service.ts:55-57`) makes git-log-based features return ONLY the HEAD commit, so "Prior PRs touching these files" shows the SAME single PR for every PR in a repo. `HistoryService.getPriorPrs` (`modules/history/service.ts:112-161`) logic is CORRECT ŌĆö `git log --follow -- <file>` per changed file (simple-git ŌåÆ `--follow`, `adapters/git/simple-git.ts:149-156`), then parse `(#N)` from commit messages (`buildPriorPrs`, service.ts:57-103) ŌĆö but a depth-1 clone contains only HEAD, so `git log` for EVERY file returns that one HEAD commit (whatever PR was squash-merged at clone time; calcom/cal.diy ŌåÆ always #27634). Confirmed: clone `rev-list --count HEAD` = 1, `git log --follow` on unrelated files both return only `0d164da ŌĆ”(#27634)`, and the live `GET /pulls/:id/prior-prs` returned only #27634. NOT seed data. `sync()` uses `RESYNC_FETCH_DEPTH=50` (`adapters/git/simple-git.ts:18-20`) but the INITIAL clone stays depth 1 ŌĆö a Refresh only helps after a resync deepens it. Fix: raise `CLONE_DEPTH`, or lazily `git fetch --deepen`/`--unshallow` when the history feature runs; also consider filtering the base-HEAD/self commit so the current PR's own base doesn't surface as a "prior PR".
- **`reconcileCiRuns` (`modules/ci/reconcile.ts`) had no per-installation error isolation** ŌĆö its `for (const installation of installations)` loop fanned out live GitHub API calls (`listWorkflowRuns` ŌåÆ octokit `listWorkflowRunsForRepo`) across EVERY `ci_installations` row in EVERY workspace, and a single installation whose repo is deleted/inaccessible/rate-limited threw a real Octokit `RequestError` (404/403/429) that aborted the ENTIRE sweep ŌĆö every other, perfectly healthy installation's runs silently failed to ingest too. Concretely surfaced via `db/seed-ci.ts`'s two demo `ci_installations` rows (`acme/payments-api`, `acme/billing-service` ŌĆö fictitious repos that exist only so the CI Runs page has fixture data to render): since `container.ts`'s `github()` always builds a REAL `OctokitGitHubClient` whenever a `GITHUB_TOKEN` secret is configured (no demo/mock bypass), and `CiRunsPage.tsx` fires `POST /ci/reconcile` on every page mount (AC-34), any dev session with a real token got a genuine "Not Found - https://docs.github.com/rest/actions/workflow-runs#list-workflow-runs-for-a-repository" toast on every CI Runs page load. Fixed by wrapping the per-installation loop body in try/catch, adding `installationsFailed` to `ReconcileSummary`, and threading an optional pino-compatible `logger` (`opts.logger`, wired from the route as `req.log`, following the `file-summary`/`why-risk-brief` services' inline-logger-type convention rather than importing `reviews/run-executor.ts`'s `Logger`) to warn-log the skip instead of throwing. Any future fan-out over installations/repos in this module needs the same per-item isolation ŌĆö a bad repo must degrade gracefully, never take down the whole workspace fleet's reconcile. The seed's fictitious demo repos were deliberately left unchanged (not worth a schema change just to flag "exclude from live reconcile") since they now just log-and-skip harmlessly on every sweep.

## Codebase Patterns

- **Eval-case delete (AC-24, L06) soft-excludes via a jsonb marker on `eval_cases.input_meta`, not a schema column.** `eval_runs.case_id` is `onDelete: 'cascade'` (`schema/eval.ts`), so a real `DELETE FROM eval_cases` would silently erase every historical `eval_runs` row that ever scored it ŌĆö the opposite of AC-24 ("prior runs that already scored it remain in history"). `EvalRepository.deleteCase` instead merges `{ _deleted: true }` into the existing `input_meta` jsonb and every "live set" read (`listCasesForOwner`, and therefore `latestRunsForOwner`'s derived aggregate) filters it out in application code (`isDeleted()` in `modules/eval/repository.ts`). No new migration needed ŌĆö `input_meta` is already jsonb. A deleted case's row (and every run row pointing at it) stays fully queryable by id (`getCaseIncludingDeleted`) for run-history joins/comparisons. Regression test: `test/eval-repository.it.test.ts` deletes a case with an existing run and asserts the run row survives while the case leaves `listCasesForOwner`.
- **`container.llm(id)` transparently prefers `ContainerOverrides.llm[id]` over the real provider** (`platform/container.ts` `llm()`: `const injected = this.overrides.llm?.[id]; if (injected) return injected;`) ŌĆö so any service that resolves its LLM via `container.llm(agent.provider)` (e.g. the eval run path, `modules/eval/service.ts` `runCase`) can be driven fully offline over REAL HTTP (`app.inject`) by passing `overrides: { llm: { openai: someMockProvider } }` to `buildApp(...)` ŌĆö no per-call injection hook needed on the route/service itself. This is how `test/eval-routes.it.test.ts` exercises the full createŌåÆrunŌåÆcompareŌåÆpromote round trip through HTTP with zero keys/network: it builds `createMockReviewerLLM(diff, 'baseline')` (T5, `modules/eval/mock-reviewer.ts`) and passes it as the container-wide `openai` override.
- **L06 eval "promote" does NOT literally set `agents.version` to the target number ŌĆö it re-applies that version's config through the normal `AgentsRepository.update()` path, which bumps to a NEW version and snapshots it.** There is no existing "activate an old version in place" primitive in the agents module (versions/`agent_versions` rows are append-only/immutable by convention ŌĆö see the `Skill body versioning`/`agent_versions` notes above), so `EvalService.promote(workspaceId, agentId, version)` looks up the target `agent_versions` snapshot, and calls `agentsRepo.update(...)` with that snapshot's fields (provider/model/system_prompt/output_schema/strategy/ci_fail_on/repo_intel) + `agentsRepo.setSkills(...)` for its skill list ŌĆö producing a new version whose CONFIG is identical to the target, not a version-number rollback. Same-version target (`agent.version === version`) short-circuits as a true no-op (no update call at all). If a future task needs a literal "reactivate version N as N" semantics, that's a new repository primitive, not something `AgentsRepository.update` gives you today.
- **The `skill.enabled` gate now has TWO independent enforcement points, not one ŌĆö grep both when changing either.** Production review composition (`run-executor.ts`, `runOneAgent`) has always filtered `linkedSkills` to `skill.enabled` before injecting a body into a live LLM call. The skill-eval run path (`modules/eval/service.ts` `runAllForOwner`'s skill branch + `runSingleCase`'s skill branch) initially did NOT mirror this ŌĆö it read `skillsRepo.getById` and injected `skill.body` regardless of `enabled`, letting an unvetted/disabled (e.g. freshly-imported, `source: 'imported_url'`) skill's body reach the LLM provider before the human-enable step, via the eval-run side door. Fixed by throwing `ValidationError` ("Skill is disabled ŌĆö enable it to run evals.", 422) in BOTH skill-run entry points the moment `!skill.enabled`, mirroring `run-executor.ts`'s filter. Deliberately scoped to RUNS only: authoring/listing/editing eval cases and reading the dashboard (`listSkillCases`, `authorSkillCase`, `skillDashboard`, `skillRunHistory`) stay unguarded for a disabled skill ŌĆö only the two paths that actually invoke `reviewPullRequest` check `enabled`. Regression tests: `test/eval-skill.it.test.ts` inserts a `enabled:false` skill and asserts `POST /skills/:id/eval-runs` and `POST /eval-cases/:id/eval-runs` both 422 while case-authoring/listing/dashboard stay 200/201.
- **The Export-to-CI feature (SPEC-05) is likewise part-pre-wired: `agent_runs.source` (`db/schema/runs.ts`, `text('source', {enum:['local','ci']}).notNull().default('local')`) and `RunTrace.config.source` (vendored `contracts/trace.ts`) already existed before any T1 work, and `run.repo.ts` `createAgentRun` already writes `source: 'local'`.** What was MISSING was `RunSummary.source` (the flat per-run-row contract read by the PR page's run history / the future agent Stats tab, AC-42) ŌĆö `listRunsForPull`'s `rows.map(...)` selects the WHOLE `agentRuns` row as `run` (so `run.source` was already sitting right there, unused) but the returned object literal never copied it out. Adding `source` to `RunSummary` therefore needed exactly ONE line in `run.repo.ts` (`source: run.source`) to keep `tsc --noEmit` green ŌĆö not a new column, not a new join. Lesson: when a shared contract gains a field that mirrors an EXISTING DB column, grep the repository's `.select({...})`/row-mapping code first ŌĆö the raw value is very often already in scope and only needs to be threaded into the returned literal, distinct from genuinely new derived/joined data.
- **T5 (`modules/ci/bundle.ts`) design decisions worth knowing before wiring T6's DB layer.** (1) `agents`/`skills` tables have **no persisted `slug` column** ŌĆö the slug is recomputed from `agent.name` on EVERY export via `SlugAllocator` (`modules/ci/slug.ts`). For AC-17 idempotency (re-export reuses the same manifest/workflow filenames), the caller (T6) MUST pass `existingSlugs` = every OTHER exported agent's slug in the workspace, **excluding this agent's own** ŌĆö recomputing from an unchanged name then naturally reproduces the same slug; only a genuinely different agent's colliding name should trigger the `-2`/`-3` suffix. - **pgvector column dimension mismatch causes silent zero-result queries ŌĆö queries return 0 rows when the embedding model changes but the stored vector column's dimension was not updated.** Symptom: a pgvector similarity search that previously returned 10 rows suddenly returns 0, with no error thrown. Root cause: changing the embedding model (e.g., OpenAI's `text-embedding-3-small` ŌåÆ `text-embedding-3-large`, dimensions 1536 ŌåÆ 3072) generates new embeddings with a different dimension, but the Postgres column definition (`vector(1536)`) is NOT updated to match. Postgres silently rejects inserts that exceed the column's declared dimension, or the similarity operator fails to match any rows (Postgres may permit the comparison but compute meaningless scores). The fix requires a schema change: create a new column with the correct dimension (e.g. `vector(3072)`), backfill embeddings from the embedding model, then DROP the old column and RENAME the new one. Add a Drizzle migration via `pnpm db:generate` after modifying the schema in `db/schema/*.ts`. Detect the problem empirically: run a known-good similarity query and verify row count; if it returns 0 when rows should match, check `pg_catalog.pg_attribute` to verify the column's type constraint ŌĆö `SELECT atttypmod FROM pg_attribute WHERE attname='embedding_column' AND attrelid='table_name'::regclass` shows the dimension in the type's modifier (e.g. 3072 = dimension). Confirmation: backfill new embeddings with the correct dimension and re-run the query ŌĆö if rows return, the dimension was the culprit. **Practical migration flow**: (1) Change the model constant/config in your service. (2) Edit `db/schema/<table>.ts` to update the column definition from `vector(oldDim)` to `vector(newDim)`. (3) Run `cd server && pnpm db:generate` ŌĆö this creates a migration that creates a new temporary column, copies the old data, drops the old, and renames the temp column. (4) Run `pnpm db:migrate` to apply it. (5) Backfill NEW embeddings: either truncate the table and re-embed from scratch, or run a backfill job that re-generates embeddings for existing rows using the new model. Until backfill completes, queries will return 0 rows for old embeddings (they're the wrong dimension) and whatever new rows have been backfilled. After backfill, queries return the expected rows again.
