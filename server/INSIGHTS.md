# server — Engineering Insights

Append-only log of non-obvious, hard-won lessons for this module (`@devdigest/api`,
includes `repo-intel`). Managed by the `engineering-insights` skill. Add each entry
under one section; keep it actionable cold; never edit or delete existing entries.

## What Works

- **Anchoring `cloneDir` to `import.meta.url` fixes the MCP vs API blast divergence with zero runtime cost.** Replace `resolve(process.cwd(), cloneDirRaw)` with `resolve(dirname(fileURLToPath(import.meta.url)), '..', '..')` (= `SERVER_ROOT`) in `platform/config.ts`. Because `import.meta.url` always resolves to the physical file on disk regardless of cwd, every process that imports `config.ts` (API at `cwd=server/`, MCP at `cwd=mcp/`, any future consumer) derives the SAME `cloneDir`. Two details: (1) the default changes from the undocumented `~/.devdigest/workspace` to `<server>/clones`, aligning with `.env.example` and README; (2) an absolute `DEVDIGEST_CLONE_DIR` is still returned verbatim — the anchor only affects relative values. Regression guard in `test/config.test.ts`: call `process.chdir(os.tmpdir())` inside the test and assert `loadConfig({}).cloneDir` is the same before and after the chdir.

- **`SimpleGitClient.ensureHistory()` lazy deepen for prior-PR history: the in-process `Set<string>` cache is LOAD-BEARING, not optional.** `git fetch --shallow-since="2 years ago"` deepens a shallow clone but the clone STILL reports `git rev-parse --is-shallow-repository = true` after the fetch (the shallow marker stays in `.git/shallow` until `--unshallow`). Without the `private deepened = new Set<string>()` cache keyed on the absolute clone path, every `log()` call — including the 25 per-file calls inside one `HistoryService.getPriorPrs` request — would re-run the `rev-parse` + `fetch` on every invocation. The cache also means offline/auth failures (fetch throws → not added to `deepened`) are automatically retried on the next request, with no extra try/catch needed. Detect shallow clones with `rev-parse --is-shallow-repository` (git ≥ 2.15); for older git, fall back to checking `existsSync('.git/shallow')`.

## What Doesn't Work

- The denormalized `agent_runs.findings_count` / `blockers` columns DRIFT from reality — they can be `0`/null while the run's review actually has findings — so a Timeline run row built from them reads "0 finding(s)" + a green "approved" badge while the Review-runs accordion below (reading `review.findings`) correctly shows the findings. `completeAgentRun` does set `findingsCount` at run end, but it's unreliable cold (stale rows, seed data, any path that adds/changes findings after completion). Don't trust those columns on read: in `run.repo.ts` `listRunsForPull` the per-severity `findings_counts` was ALREADY computed fresh from `findings → reviews.run_id`, but `findings_count`/`blockers` were still read from the denorm columns — derive those two from the same fresh join instead (the client `RunHistory.outcomeOf` keys the badge off `findings_count`+`blockers`, so this also fixes the wrong "approved" color with no client change). Regression test pattern: insert a run with `findingsCount: 0, blockers: null` but a review with a CRITICAL + WARNING finding, then assert `GET /pulls/:id/runs` returns `findings_count: 2, blockers: 1` (`findings-counts.it.test.ts`).
- The demo seed (`server/src/db/seed.ts`) creates PR #482's sample review DECOUPLED from the timeline run: it inserts the `review` (+ findings) with NO `run_id` and NO `agent_id`, then later inserts the `agent_runs` row separately (only `findings_count`/`blockers` are denormalized onto it). So any feature that attributes a run's data via `reviews.run_id` (e.g. per-run `findings_counts` in `run.repo.ts` `listRunsForPull`, which joins `findings → reviews ON reviews.run_id = agent_run.id`) gets NOTHING for seeded runs → null → the UI shows the em-dash fallback. Real runs are fine (`run-executor` sets `review.runId`); only the seed is unrealistic. Fix: after creating the seeded run, `db.update(t.reviews).set({ runId, agentId })` for that PR's review — place it OUTSIDE the `!existingRun` guard so re-running `pnpm db:seed` repairs an already-seeded DB (idempotent). Linking `agent_id` also fixes the Review-runs row showing a generic "Agent" instead of the agent name.
- Adding a new contract field as `.nullable()` (REQUIRED, value-or-null) instead of `.nullish()` (optional) silently breaks existing tests that build the object literally. Concretely: making `RunSummary.findings_counts` `.nullable()` failed `contracts.test.ts`'s `RunSummary.parse({...})` fixture and the client `run()`/`pr()` test builders with a Zod "Required" error, because those omit the new key. Fix: either use `.nullish()`, or update every literal fixture/builder to include the field. (`PrMeta` used `.nullish()` and needed no fixture change.)
- `GitClient.defaultBranch()` (`adapters/git/simple-git.ts`) has a three-step detection chain: (1) `git rev-parse --abbrev-ref origin/HEAD` → fails silently when `origin/HEAD` is not set on the remote/shallow clone; (2) `git rev-parse --abbrev-ref HEAD` → returns the literal string `'HEAD'` (not a branch name) whenever the clone is in detached HEAD state; (3) hardcoded fallback `'main'` → this fires for `master` repos in the detached-HEAD + no-origin/HEAD combination, silently writing `'main'` to the DB. Fix added to `defaultBranch()`: before returning `'main'`, try `git ls-remote --symref origin HEAD` and parse with `/^ref:\s+refs\/heads\/(\S+)\s+HEAD/m` — this queries the remote authoritatively and is acceptable inside a clone-job context. Any repo already in the DB with the wrong `'main'` default needs a Refresh to re-run `runCloneJob` and re-detect.
- `repos.default_branch` defaulted to `'main'` in the schema (`db/schema/repos.ts`) and was NEVER fetched from the real repo — `RepoRepository.insert` didn't set it. This silently broke TWO things for any repo whose real default is `master` (e.g. `actualbudget/actual`, `Staffjoy/v2`, `tw93/Maple`): (1) the client GitHub blob deep-links (`ConventionsWorkspace` `repoRef = activeRepo.default_branch` → `githubBlobUrl`) 404'd, and (2) repo-intel "re-analyze" (`repo-intel/service.ts` `sync(ref, repo.defaultBranch)` → `git reset --hard origin/main`) silently failed. Fix: `runCloneJob` (`modules/repos/service.ts`) now calls the new `GitClient.defaultBranch(repo)` (impl: `git rev-parse --abbrev-ref HEAD`, since `git clone` without `--branch` checks out the remote default) and persists it via `updateClonePath(repoId, path, defaultBranch)`. KEY GOTCHA: existing repo rows keep `'main'` — they only backfill on a **Refresh / re-clone** (the conventions page "Re-scan" runs `extract`, NOT a clone, so it does NOT fix the branch). When debugging a stale `default_branch`, re-clone the repo, don't re-scan.
- The `GET/PUT /settings` routes need NO route-schema change to add a brand-new settings
  key (e.g. `root_folders`, T4). `Settings`/`SettingsUpdate` (`vendor/shared/contracts/
  platform.ts`) are built as `SettingsKnown.passthrough()` (not `.strict()`), so a Zod
  object's `.partial()` on a passthrough schema stays passthrough — arbitrary extra keys in
  the `PUT /settings` body sail through validation untouched, get looped over in
  `Object.entries(body)` and upserted, and `GET /settings` returns the *raw* folded rows
  (`rowsToSettings`, no `.parse` at all) — so an unvalidated new key round-trips end-to-end
  with zero route code changes. This means "does the route need a schema change to accept
  key X" is the WRONG question for this endpoint; the real design decision is whether X's
  *shape* should be validated at write time (like `feature_models: z.record(...)` in
  `SettingsKnown`, which does add real structural validation) or left permissive and
  validated only at the READ site via a dedicated `resolveX`/`getX` helper (like
  `getRootFolders` in `modules/settings/root-folders.ts`, which `safeParse`s and falls back
  to a default on garbage). Prefer the read-site-validates pattern for anything read by a
  narrow set of consumers (here: the documents-discovery service) — it avoids touching the
  vendored contract (`*/src/vendor/shared` in three places) for a field only one caller needs.
- The Conventions Extractor has NO `replaceAll`/delete step, so re-scanning a repo ACCUMULATES duplicate candidates. `ConventionsService.extract()` (`modules/conventions/service.ts`) generates a fresh `runId`, calls `ConventionsRepository.insertMany()`, and NEVER deletes prior rows; `listByRepo()` returns every non-`rejected` row across ALL runs (it does not filter to the latest `runId`). So each "Re-scan" inserts a whole new batch of `pending` candidates on top of the old ones — the UI fills with duplicates and previously-accepted rules get re-proposed. Note there is no `server/src/modules/conventions/extractor.ts` despite the name: orchestration lives in `service.ts`, the pure steps in `reviewer-core/src/conventions/{extract,verify,assemble}.ts`. To make re-scan idempotent, delete the repo's prior non-accepted candidates (or all rows for older `runId`s) inside `extract()` before `insertMany`, or add a real `replaceAll` to the repository.
- Blast Radius reports `0 endpoints · 0 cron` for Next.js App-Router PRs (e.g. cal.com `apps/web/app/api/cron/**/route.ts`) because of TWO compounding gaps. (1) `extractEndpoints`/`extractCrons` (`adapters/codeindex/extract.ts`) only match Express/Fastify shapes (`app.get('/x')`, `router.post(...)`, `route({method,url})`) and cron-expression/`register('kind')` calls — they NEVER recognize `export async function GET|POST(...)` in a `route.ts`, where the path comes from the FILE PATH (`app/api/cron/foo/route.ts` → `/api/cron/foo`). So `file_facts` is empty for those files. (2) Even with facts present, `tryPersistentBlast` (`repo-intel/service.ts`) reads `getFileFacts(repoId, callerFiles)` — CALLER files only. A route handler IS the endpoint and has 0 cross-file callers (the framework invokes it via file-based routing), so `callerFiles` is empty and its OWN facts are never read. To surface route endpoints you must BOTH (a) make `extractEndpoints`/`extractCrons` path-aware (take `relPath`, derive the route from the path, emit one entry per exported verb, flag `/api/cron/**`) — then REINDEX, since `file_facts` is built at index time, not read time — AND (b) attribute the changed files' own facts, not just their callers'. (This is distinct from the already-documented `getResolvedCallers` precision-over-recall "0 callers is expected" note in `client/INSIGHTS.md`.)
- Blast Radius shows **0 callers for default-exported-by-separate-statement symbols** (e.g. cal.com `getAppCategories` in PR #29558), even though they have real cross-file callers AND `references` rows exist. Root cause is in `parseSymbols` (`adapters/astgrep/index.ts`): the `exported` flag is set by `unwrapExport` (handles INLINE forms — `export default function/class`, `export const x`) and a re-export pass (handles `export { x }` clauses ONLY). NEITHER handles `export default <identifier>`: for `const getAppCategories = (...) => {…}` followed by a SEPARATE `export default getAppCategories;`, the `export default` node's payload is a bare `identifier`, `handleDecl` ignores it, and the original `const` keeps `exported: false`. Proven empirically: `parseSymbols` gives `exported:true` for `export default function`/`export const arrow`, but `exported:false` for `const X=…; export default X;`. That one flag defeats BOTH caller paths: edge resolution (`resolveReferences` SQL joins `… AND s.exported = true`, `repository.ts:412`) AND the Tier-1 name-unique fallback (`getUniqueExportFiles` filters `eq(t.symbols.exported, true)`, `repository.ts:619`). Tell-tale in the UI: inline `export default function Foo()` symbols DO resolve callers while `const X=…; export default X` siblings show 0. This is the most common export form in Next.js/React monorepos, and it silently defeats the `blast-radius-caller-resolution-gaps` plan's own Tier-1 fallback (the plan chased `file_edges`/cross-package/type-refs and missed this). Fix: back-patch `exported` for `export default <identifier>` in `parseSymbols` (mirror the `export { X }` re-export pass: an `export_statement` with a `default` child + a bare `identifier` payload → mark the prior same-named symbol exported), add a regression case to `test/astgrep.test.ts`, and bump `INDEXER_VERSION` (`repo-intel/constants.ts`) — `symbols.exported` is written at index time so existing repos need a reindex. Residual limitation (separate, deeper): a default import bound to a DIFFERENT local name (`import gac from "…/getAppCategories"; gac()`) records the reference as `gac`, so it still won't match the declared `getAppCategories`.
- Blast Radius / repo-intel reads resolve the clone root in a **cwd-dependent** way that DISAGREES with the absolute path stored in the DB — so the SAME `BlastService.getBlast` returns symbols in the API but `no_data`/0 symbols in the MCP server (or any script). `RipgrepCodeIndex.symbols()/references()/grep()` (`adapters/codeindex/ripgrep.ts:46-126`) compute the root via `git.clonePathFor(repo)` = `join(config.cloneDir, owner, name)` (`adapters/git/simple-git.ts:37-39`); they IGNORE `repos.clone_path` (the absolute path written at clone time, also loaded as `repo.clonePath` in `getBlastRadius` and used by `readClone`). `config.cloneDir` (`platform/config.ts:64-80`) resolves a relative `DEVDIGEST_CLONE_DIR` against `process.cwd()` (line 68), and `import 'dotenv/config'` (line 1) loads `.env` from cwd ONLY. Result: API (cwd=`server/`, `server/.env` has `DEVDIGEST_CLONE_DIR=./clones`) → `server/clones/<owner>/<name>` ✓; MCP server (cwd pinned to `mcp/` by `bin/devdigest-mcp.mjs`, no `mcp/.env`, `.mcp.json` passes only `DATABASE_URL`) → falls back to the default `~/.devdigest/workspace` (which doesn't exist) → the directory walk finds nothing → `{changedSymbols:[], degraded:true, reason:'no_data'}`. Proven by A/B with identical bootstrap code: 0 symbols under MCP defaults vs calcom/cal.diy#29557→2, #28926→70 once `DEVDIGEST_CLONE_DIR` pointed at `server/clones` (matching the live `GET /pulls/:id/blast` exactly). Robust fix: read the persisted `repo.clonePath` instead of re-deriving via `clonePathFor`, or anchor relative `cloneDir` to a stable dir (not `process.cwd()`) AND align the default (`./clones` vs `~/.devdigest/workspace` disagree). Affects every cwd-sensitive consumer (MCP, cron, ad-hoc scripts), not just blast.
- `CLONE_DEPTH = 1` (`modules/repos/constants.ts:9`, used in `modules/repos/service.ts:55-57`) makes git-log-based features return ONLY the HEAD commit, so "Prior PRs touching these files" shows the SAME single PR for every PR in a repo. `HistoryService.getPriorPrs` (`modules/history/service.ts:112-161`) logic is CORRECT — `git log --follow -- <file>` per changed file (simple-git → `--follow`, `adapters/git/simple-git.ts:149-156`), then parse `(#N)` from commit messages (`buildPriorPrs`, service.ts:57-103) — but a depth-1 clone contains only HEAD, so `git log` for EVERY file returns that one HEAD commit (whatever PR was squash-merged at clone time; calcom/cal.diy → always #27634). Confirmed: clone `rev-list --count HEAD` = 1, `git log --follow` on unrelated files both return only `0d164da …(#27634)`, and the live `GET /pulls/:id/prior-prs` returned only #27634. NOT seed data. `sync()` uses `RESYNC_FETCH_DEPTH=50` (`adapters/git/simple-git.ts:18-20`) but the INITIAL clone stays depth 1 — a Refresh only helps after a resync deepens it. Fix: raise `CLONE_DEPTH`, or lazily `git fetch --deepen`/`--unshallow` when the history feature runs; also consider filtering the base-HEAD/self commit so the current PR's own base doesn't surface as a "prior PR".

## Codebase Patterns

- The Skills feature (L02) was ~90% pre-wired before any module existed: tables `skills`/`skill_versions`/`agent_skills` (the join has an `order` col), shared contracts `Skill`/`AgentSkillLink`/`AgentVersionConfig.skills`, the reviewer-core prompt slot (`assemblePrompt` `parts.skills` → `## Skills / rules`), and `RunTrace.prompt_assembly.skills`. The ONLY missing server wire was `run-executor.ts` `runOneAgent`: load `this.agents.linkedSkills(agent.id)`, filter to `skill.enabled`, pass the bodies as `skills:` to `reviewPullRequest`. Lesson: before planning new tables/fields for a lesson, grep `db/schema/*`, `vendor/shared/contracts/*`, and `reviewer-core/src/prompt.ts` — the empty tables + optional prompt slots are usually already there.
- Skill body versioning mirrors agent config versioning but keys on the BODY only: `modules/skills/repository.ts` `update()` bumps `version` + appends a `skill_versions` snapshot ONLY when `isBodyChange(existing, patch)` is true (`modules/skills/helpers.ts`). Editing name/description/type or toggling `enabled` does NOT version. The seed writes the v1 snapshot inline (mirror `SkillsRepository.insert`'s `snapshotBody`) — don't forget it or the Versions tab shows nothing for seeded skills.
- SUPERSEDES the "draft until first save (v0)" note in *Recurring Errors & Fixes*: the v0-draft model was REMOVED. `SkillsRepository.insert` now persists at `INITIAL_SKILL_VERSION` (1) AND writes a v1 snapshot immediately; `update()` just bumps on a real body change (no more `commitDraft`/`isDraft` branch; `DRAFT_SKILL_VERSION` constant deleted). The "nothing persisted until Save" behavior moved to the CLIENT: the blank-create button (`SkillsWorkspace`) opens an in-memory draft (`ConfigTab` create mode) and only POSTs on Save → so create=v1, next body edit=v2. Import drawer + conventions `CreateSkillModal` still POST directly (their content is final at confirm) and are therefore v1 immediately.
- Skill names are UNIQUE per workspace, case-insensitively: unique index `skills_ws_name_uq` on `(workspace_id, lower(name))` (migration `0012`, which first de-duplicates any pre-existing names by suffixing " 2"/" 3" so the index can be created). `SkillsService.create`/`update` pre-check via `SkillsRepository.findByName` (case-insensitive) and throw `ConflictError` (new 409 in `platform/errors.ts`) — caught client-side as `ApiError` (status 409) and shown inline by the name field. The client auto-suffixes the auto-generated default name (`uniqueName` in `client/src/lib/unique-name.ts`) so the common "Add Skill" / "Create skill from conventions" flow never trips the 409.
- In `run.repo.ts` `listRunsForPull`, derive ALL of `findings_count`, `blockers`, AND `findings_counts` from the SAME fresh `findings → reviews.run_id` join (`countsByRun`) — do NOT read `findings_count`/`blockers` from the denormalized `agent_runs` columns (see the matching "What Doesn't Work" note on why they drift). `findings_count = c.critical+c.warning+c.suggestion`, `blockers = c.critical`, falling back to the run's stored column only when `countsByRun` has no entry (failed/cancelled runs with no review). This keeps the Timeline badge/count and the Review-runs accordion (which reads `review.findings`) showing the same numbers.
- Vendored shared contracts are **edited directly per-package**, not generated. `server/src/vendor/shared/contracts/*.ts` and `client/src/vendor/shared/contracts/*.ts` are NOT byte-identical (comments/imports differ) and there is no upstream `shared/` dir or re-vendor script despite CLAUDE.md saying "edit upstream, then re-vendor". When adding a field (e.g. `cost_usd` on `RunStats`/`RunSummary`/`PrMeta`), edit each package's copy by hand and keep the field sets in sync. Precedent: commit `d45ab0d` did exactly this. reviewer-core has no vendored copy — it imports the types via the `@devdigest/shared` alias.
- `GET /repos/:id/pulls` (`server/src/modules/pulls/routes.ts`) only syncs PRs from GitHub when `container.github()` resolves; with no token configured it throws and the route silently serves persisted PRs. In integration tests that DON'T pass a `github` override (e.g. `reviews.it.test.ts`'s `appWith`), the list returns exactly the DB-inserted PRs — safe for asserting computed columns (score, `cost_usd` aggregate) without GitHub-import noise.
- Per-severity findings counters (PR list FINDINGS column + Agent-runs rows) are computed ON READ, not denormalized onto `agent_runs`. Reuse `rollupSeverities` / `groupSeverities` from `server/src/modules/pulls/status.ts` (pure, unit-tested). PR list: in `pulls/routes.ts`, alongside the latest-review-per-PR score loop, also collect the latest review id PER `(prId, agentId)` and tally their findings → aggregates across reviewer agents (matches the multi-agent detail view); SCORE stays single-latest-review. Runs: in `reviews/repository/run.repo.ts` `listRunsForPull`, join `findings → reviews` on `reviews.run_id IN (runIds)`. Mirrors the existing latest-review-score read pattern and needs NO migration. Counts INCLUDE dismissed findings (kept consistent with the run's `findings_count`/`blockers`).
- Backend layering today is ports/adapters + a single composition root (`server/src/platform/container.ts`), but it is NOT full onion. Two non-obvious facts: (1) the port interfaces (`LLMProvider`, `GitClient`, `GitHubClient`, `Embedder`, `SecretsProvider`) live in vendored `@devdigest/shared`, NOT next to their concrete classes in `src/adapters/*` — to find a port's contract, read `shared`, not the adapter file; (2) there are no domain entity classes or use-case DTOs — a module's `repository.ts` returns Drizzle `$inferSelect` rows straight out and `service.ts` orchestrates against the whole `Container`. `reviewer-core/src` is the only pure inner core. The `backend-onion-architecture` skill (`.claude/skills/`) documents the target + a module-by-module migration (start with `reviews`, the richest logic); don't try to retrofit all modules at once.
- `RepoIntel.getConventionSamples(repoId, n)` (`repo-intel/service.ts`) is `getTopFilesByRank` under the hood, and `isJunkPath` deliberately DROPS config files — the `JUNK_PATH_PATTERNS` list includes `.config.`, `eslint`, `prettier`, `vitest.`, `/migrations/`, `.d.ts`, tests. So eslint/tsconfig/prettier/package.json will NEVER come back from `getConventionSamples`, even though they're the highest-signal source of house conventions. If a feature wants them (e.g. Conventions Extractor sampling), read them DIRECTLY from the clone via `fs.readFile(join(clonePath, name))` against an explicit allowlist — `getConventionSamples` returns only ranked NON-config source files. Also: it returns `[]` (not throws) when `config.repoIntelEnabled` is false or the repo has no `file_rank` rows (un-indexed), so the sampler must tolerate an empty list.
- `RunLogger` (`src/platform/run-logger.ts`) has NO `warn` method — `RunEventKind` is only `info | tool | result | error` (see the `LEVEL` map). To surface a non-fatal warning in the Live Log, call `runLog.info('⚠️ …')`; do NOT use `runLog.error(...)` — an `error` event styles the run as failed in the UI/SSE stream. (Used for the flash-tier-model gate warning in `modules/reviews/run-executor.ts`.)
- `MockLLMProvider` (`src/adapters/mocks.ts`) returns the SAME fixture on every `completeStructured` call by default — so a test of a multi-CALL flow (the review re-sample, or any retry/dialogue) can't distinguish call 1 from call 2 with `structured`/`structuredBySchema` alone. Use the `structuredSequence: [fixtureA, fixtureB, …]` option (Nth call → Nth entry; repeats the last once exhausted) to drive distinct responses per call. For flows where DIFFERENT schemas are called in the same request (e.g. `classifyIntent` uses schemaName `'Intent'` and `reviewPullRequest` uses `'Review'`), `structuredBySchema: { Intent: INTENT_FIXTURE, Review: REVIEW_FIXTURE }` is cleaner than `structuredSequence` — it routes by name, not call order. An omitted key falls back to `{}`, which typically fails the schema and throws — making it a zero-extra-code way to simulate "LLM fails for this schema only" in non-fatal error path tests.
- To simulate "GitHub unavailable" without subclassing `MockGitHubClient`, inject `secrets: new MockSecretsProvider({})` (no `GITHUB_TOKEN`) and omit the `github` override. `container.github()` then throws `ConfigError('GITHUB_TOKEN is not configured')`, which the intent service's try/catch catches and treats as a degraded-mode condition — classifying from stored PR data + empty files.
- Services that depend on `Container` (e.g. `IntentService`) can be instantiated directly from a test using `new IntentService(app.container)` after `buildApp(...)`. This gives access to the full return value (e.g. `result.savedTokens`) without going through HTTP, while still using the real DB and injected mock adapters. Useful when the route intentionally returns a subset of the service result.

- Skills routes resolve the DEFAULT workspace via `getContext(app.container, req)` — there is NO workspace header/param on `/skills/*` GET routes. Consequence for `*.it.test.ts`: a route-level `app.inject('/skills/:id/stats')` only finds a skill that lives in the SEEDED default workspace; a skill you hand-build in a fresh non-default workspace returns 404 through the route (the handler looks it up in the default ws). To assert aggregation over isolated hand-built data, call `new SkillsRepository(db).getStats(workspaceId, id, windowDays)` directly (deterministic denominator, no seed noise) and test the HTTP route separately against the seeded skill + a random-uuid 404. Pattern used in `test/skill-stats.it.test.ts`.
- Per-skill stats (`SkillsRepository.getStats`, `GET /skills/:id/stats`) join `findings → reviews → agent_skills → skills` via `reviews.agent_id` (NOT via `agent_runs`); `reviews` carries `agentId` + `createdAt` + `prId` in one place, so it's the single join base for used-by / pull-frequency / accept-rate / findings. The demo seed's only standalone review (PR #482) is attributed to the Security Reviewer, which uses few skills, so most skills would show ALL-ZERO stats. Fix that made the Stats tab demoable: a dedicated demo PR (#501, added in `db/seed.ts` + `STATS_DEMO_REVIEWS` in `db/seed-skills.ts`) whose reviews are attributed to the rubric's agents with categorized + decided (`accepted_at`/`dismissed_at`) findings. Kept on its OWN PR so PR #482 (asserted by existing tests + e2e flows) stays untouched. Lesson: when a per-skill/per-agent metric reads `reviews.agent_id`, the base seed won't exercise it — add attributed reviews on a separate PR rather than mutating #482.
- There is NO per-finding or per-review cost in the schema, so any "$ per finding / per finding-category" metric is infeasible without a schema + cost-attribution change — don't try to wire it. `cost_usd` exists ONLY at run granularity: `agent_runs`, `eval_runs`, `ci_runs` (all `db/schema/*`); `findings` (`db/schema/reviews.ts`) and `reviews` have no money column. A run emits many findings across many categories under one lump sum, so cost can't be split per finding/category without inventing a number. This is why the Skills Stats "Findings by category" donut shows COUNTS, not dollars — the client deliberately passes `valuePrefix=""` to the vendored `Donut` (whose default is `"$"`) for exactly this reason. So "show price per category" is a correct-by-design count, not a bug.

- `ReviewRepository.reviewsForPull` (and its underlying `review.repo.ts` query) returns ALL findings for each review — **including dismissed ones** (`dismissedAt IS NOT NULL`). Any feature that surfaces "active" findings (SmartDiff line badges, annotation counts, etc.) must guard with `if (finding.dismissedAt != null) continue;` in its own aggregation loop. The field is always present on `FindingRow` (`typeof t.findings.$inferSelect` includes `dismissedAt`), so no extra join or schema change is needed — just the null-check. Without it, dismissing a finding in the Agent runs tab leaves the file badge count and per-line pill unchanged in the Files-changed view, confusing users who think dismissal did nothing. ## Recurring Errors & Fixes

- When importing a Zod schema from `@devdigest/shared` to call `.safeParse()` (or `.parse()`) at runtime, it MUST be a value import, not a type-only import. `import type { PrBrief }` strips the import to zero bytes at compile time — calling `PrBrief.safeParse(row.json)` at runtime throws `TypeError: PrBrief.safeParse is not a function`. Fix: mix the import — `import { PrBrief, type Intent }` — so the schema object is present at runtime while type-only symbols are erased. TypeScript will NOT catch this at compile time because `import type` silently satisfies type-checking; the error only surfaces at runtime. Applies to any repo helper that validates DB data with a shared Zod schema (e.g. `getBrief` in `pull.repo.ts`).

- An `.it.test.ts` that inserts its OWN fully-linked fixtures can stay green while the feature is broken for real users, because the demo seed (which users actually run via `pnpm db:seed`) builds the data DIFFERENTLY. Real case: `findings-counts.it.test.ts` set `reviews.run_id` on its hand-built rows, so per-run `findings_counts` passed — but the seed left `run_id` null, so the live app showed "—". Fix/guard: when a feature READS seed-shaped data, add a test that runs against `seed()` output (call `seed(db)` then query the seeded entity, e.g. PR #482) — not just hand-built rows. The two fixture shapes diverge; test both.

- "Replace the whole set" repository methods (`agents/repository.ts` `setSkills`: delete the agent's links + re-insert) break under concurrent `POST /agents/:id/skills`. ROOT CAUSE of the concurrency: the vendored `Checkbox` (`client/src/vendor/ui/kit/Checkbox.tsx`) is a `<button>` inside a `<label>`, so one click fires `onChange` TWICE (the label re-dispatches activation to its labeled control) → two identical `setSkills` mutations land at once. The failures escalated through THREE attempts — learn from the dead ends: (1) plain non-atomic delete+insert → `duplicate key value violates unique constraint "agent_skills_agent_id_skill_id_pk"`. (2) wrapping in `this.db.transaction(...)` alone → STILL duplicate-keys: under PostgreSQL READ COMMITTED the 2nd tx's DELETE can't see the 1st tx's just-committed INSERTs. (3) `delete … notInArray(skillId, ids)` + multi-row UPSERT (`onConflictDoUpdate`) → fixes the duplicate-key but then `deadlock detected`: two txns lock agent_skills rows/index entries in opposite order. THE FIX THAT WORKS: serialize per-agent with a transaction-scoped advisory lock at the top of the tx — `await tx.execute(sql\`SELECT pg_advisory_xact_lock(hashtext(${agentId}))\`)` — then a plain delete-all + insert is correct (auto-released at COMMIT/ROLLBACK; concurrent same-agent writers run one-at-a-time). Still dedupe ids (`[...new Set]`). Verify with TRULY CONCURRENT requests (`Promise.all` of N identical `setSkills`, or `curl … &` ×N, asserting no `deadlock`/`duplicate key`) — a SEQUENTIAL `await`-each test passes against ALL the broken versions and proves nothing. Since `src/vendor/**` is "do not touch", the authoritative fix is server-side serialization, not the Checkbox. (`skills.it.test.ts` has the concurrent-burst regression.)
- Skill body versioning is "draft until first save" (chosen product behavior): `skills/repository.ts` `insert()` creates a skill at `version: DRAFT_SKILL_VERSION` (0) with NO `skill_versions` snapshot — the auto-scaffold body must not burn v1. `update()` COMMITS a draft to v1 on the first save that includes a body (even if the body still equals the scaffold: `isDraft && patch.body !== undefined`), and only AFTER that does a real body change bump v2+. So create→v0, first save→v1, next edit→v2. Applies to ALL creates incl. import (an imported skill is a v0 draft until saved). The seed is the exception — it writes skills via `db.insert(t.skills)` directly (DB default `version` = 1) + an inline v1 snapshot, so seeded skills are committed v1. - **Never interpolate a JS array into a raw Drizzle `sql\`\`` with `= ANY(${arr})` — it throws `PostgresError: malformed array literal: "<value>"` at runtime.** Real case: `getUniqueExportFiles` (`repo-intel/repository.ts`) was written as `sql\`… WHERE name = ANY(${names}) GROUP BY name HAVING count(distinct path)=1\``.
